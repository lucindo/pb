**Gathered:** 2026-05-10

<domain>
## Phase Boundary

Phase 5.1 ships two carry-forward fixes from the Phase 5 manual UAT log (`05-04-UAT-LOG.md` Findings 1+2) and one Phase 2 visual carry-forward, all on real-device Safari surfaces:

1. **iOS Safari audio-resume on visibilitychange→visible.** When the user manually locks the phone mid-session and unlocks ≥30 s later, the AudioContext is left in a `suspended` state by iOS Safari and never resumes — audio cues stay silent for the rest of the session even though the wake-lock has re-acquired and the visual orb continues. Phase 5.1 mirrors the Phase 5 D-03 visibility-re-acquire pattern inside `useAudioCues` so the AC `resume()` path is symmetric with the wake-lock path: same listener placement, same ref-gated guard, same silent-fallback posture.

2. **Safari desktop breathing-orb max-scale visual gap.** At peak inhale the inner orb (`transform: scale(MAX_SCALE = 1.0)`) does not visually reach the dashed outer reference ring. The gap is sub-pixel/box-sizing arithmetic on Safari only — Chromium renders both edges coincident. Phase 5.1 compensates the outer ring sizing (negative inset OR explicit content-box width/height) without touching the orb scale constants or the Phase 2 reduced-motion contract.

The phase boundary is fixed by `.planning/ROADMAP.md` §Phase 5.1. **Out of scope:** Android Chrome S2 carry-forward UAT (tracked separately in `05-04-UAT-LOG.md` Gap 1 — closed by physical-device run, not by code work in this phase), MOBL-03 user-visible Wake Lock fallback explanation (v2), pause/resume mid-session (v2 SESS-06), volume control (v2 AUDI-03), Forrest learning content (Phase 6), any change to Phase 1 timing math, Phase 3 dual-anchor / scheduleNextCue contract, Phase 4 storage envelope, or Phase 5 useWakeLock surface.

<decisions>
## Implementation Decisions

### Audio Resume Site
- **D-01:** The `visibilitychange` listener that drives audio resume is **owned by the `useAudioCues` hook**, mirroring Phase 5 D-04 (`useWakeLock` owns its own visibility listener). Listener installed in a `useEffect([])` on mount, removed in cleanup. Ref-gated guard inside the handler. Symmetry with the wake-lock pattern is the primary reason — future readers see the same shape twice and the pattern becomes obvious.
- **D-02:** Listener placement is in the **hook**, not in `audioEngine.ts`. The engine stays React-free / DOM-event-free (current invariant — engine module currently has zero `document.*` and zero `window.*` access). The hook is the React-side resource owner; the engine is the pure-imperative service composed by it. This is the same boundary Phase 3 D-09 → D-11 / Phase 5 D-11 → D-13 already established.

### Audio Resume Gating
- **D-03:** The visibility handler calls `audioCtx.resume()` **whenever an engine exists** (`engineRef.current !== null`). Mirrors Phase 5 D-04's `wasAcquiredRef`-gated guard pattern. Covers running and lead-in (any time the AudioContext was constructed and not yet closed). The hook does NOT need to know about App-level `appPhase` — the `engineRef` is a sufficient and self-owned signal.
- **D-04:** Gate is **`engineRef.current !== null`** specifically (not `audioCtx.state === 'suspended'`). The state read is unreliable on iOS during the display-sleep window and adds a polling-feel. The ref tells us "we own a live AC"; calling `resume()` on an already-running AC resolves silently — same idempotent posture as Phase 5 D-08.

### Make-Up Cue Policy
- **D-05:** After a successful resume mid-phase, **wait for the next phase boundary** — do NOT fire a make-up cue. Mirrors Phase 3 D-08 (unmute waits for boundary). The next In/Out cue fires from the existing `scheduleNextCue` dispatch driven by App.tsx Plan 03-04's boundary effect — no make-up scheduling code needed. Avoids audible "late strike" that would feel like a glitch and avoids touching the Plan 03-04 dual-anchor invariant (Phase 3 Pitfall 2).
- **D-06:** No re-anchoring of `audioAnchorRef` on resume. The original anchor captured at lead-in completion stays valid — the audio clock and the breathing plan offsets are unchanged across the suspend/resume cycle. Re-anchoring would cascade into App.tsx integration tests and risks misaligning subsequent cues.

### Engine Surface
- **D-07:** `audioEngine.ts` gains a single new method: **`resume(): Promise<void>`** that calls `audioCtx.resume()` inside a try/catch. Idempotent: calling `resume()` on an already-running AC resolves silently. No state read inside (mirrors D-04's posture — the caller decides whether to call). Failures absorbed silently per D-09 below. This is the minimal engine API change needed; everything else stays internal to the hook.
- **D-08:** `useAudioCues` gains no new public method. The visibility listener is fully internal — exactly the same pattern as `useWakeLock`'s D-12 (no `isAcquired` / no `isSupported` / no resume status exposed). The public hook surface (`status`, `audioAvailable`, `muted`, `start`, `stop`, `setMuted`, `notifyPhaseBoundary`, `audioNow`) is unchanged. App.tsx requires zero edits beyond the test files.

### Failure Posture
- **D-09:** **All resume failures are silently absorbed.** Concretely: `audioCtx.resume()` rejecting (browser veto, AC already closed, race against unmount), the listener firing after `engineRef.current` was nulled by `stop()`, any synchronous throw — all resolve with no user-facing artifact. No banner, no toast, no console warning required (single dev-only `console.debug` is acceptable but optional). Matches Phase 3 D-10 (AC fail → visuals-only) and Phase 5 D-09 (Wake Lock fail → silent). The session continues; if iOS rejects resume, the user is no worse off than the current Phase 5 baseline.

### Orb Scale Fix
- **D-10:** **Outer reference ring sizing compensated; orb scale constants UNCHANGED.** Concretely: in `BreathingShape.tsx` the outer ring `<span>` (currently `className="orb-ring--outer absolute inset-0 rounded-full border-solid"`) is updated so its visible edge matches the orb at `scale(MAX_SCALE = 1.0)` on Safari. Two acceptable mechanisms (planner picks based on cross-browser test):
  1. Explicit width/height: 100% with `box-sizing: content-box` on `.orb-ring--outer` so the 1.5 px border lives outside the 100% box, OR
  2. Negative inset compensation: replace `inset-0` with explicit `inset: -1.5px` (or the equivalent left/top/right/bottom -1.5px) so the border-box's outer edge sits 1.5 px beyond the container's 100% box.

  Neither approach changes `--orb-scale-min` / `--orb-scale-max` / `--orb-scale-mid` or the corresponding TS constants. The TS↔CSS contract documented in `BreathingShape.tsx:18-22` and `theme.css:14-20` (IN-01 sync rule) is preserved.
- **D-11:** **Inner reference ring is NOT touched.** UAT only flagged the outer ring (`05-04-UAT-LOG.md` Finding 1: "inner aqua orb does not visually meet the dashed outer guide ring"). The inner ring is already explicitly sized via `width: ${MIN_SCALE * 100}%` / `height: ${MIN_SCALE * 100}%` (`BreathingShape.tsx:80-83`) so it does NOT share the `inset-0` + border-box arithmetic that produces the outer-ring gap. Don't fix what isn't broken.
- **D-12:** **Lead-in render path receives the same outer-ring fix.** `BreathingShapeLeadIn` (`BreathingShape.tsx:126-171`) renders its own copy of the outer ring at `inset-0 + border-solid`. The fix applies to both render sites or the gap reappears during the 3-2-1 countdown.

### Reduced-Motion Compatibility
- **D-13:** **The Phase 2 reduced-motion contract is preserved verbatim.** `MID_SCALE = 0.79` is the average of `MIN_SCALE` and `MAX_SCALE`; if the planner had bumped MAX_SCALE in D-10 it would have shifted MID_SCALE and broken `theme.css:18-20` IN-01 sync + the Phase 2 D-06 fixed-mid-scale contract. Compensating the outer ring instead leaves all three scale constants and the reduced-motion render path untouched. The `@media (prefers-reduced-motion: reduce)` block in `theme.css:112-117` is unchanged.

### Test Infrastructure
- **D-14:** **`vitest.setup.ts` FakeAudioContext is extended with `state` + `resume()` + `_simulateSuspend()`.** Following Phase 5 D-13 precedent (FakeWakeLock added the same way). Concrete additions:
  - A `state` getter that returns `'running'` by default and `'suspended'` after `_simulateSuspend()` is called.
  - A `resume()` method that flips `state` back to `'running'` and resolves. Spy-able with `vi.spyOn(audioCtxInstance, 'resume')`.
  - A `_simulateSuspend()` non-standard hook (underscore-prefixed, test-only) that sets `state='suspended'` so tests can model the iOS lock-screen window.
  - The existing FakeAudioContext's other behaviors (currentTime, destination, close, scheduleTick/Cue) remain unchanged.
- **D-15:** **Hook tests dispatch a `visibilitychange` event** via `Object.defineProperty(document, 'visibilityState', { value: 'visible' })` + `document.dispatchEvent(new Event('visibilitychange'))`. Same pattern as Phase 5 D-13 wake-lock visibility tests in `useWakeLock.test.tsx`. Tests assert: (a) resume() is called when visibility→visible AND engineRef exists, (b) resume() is NOT called when engineRef is null (after stop()), (c) silent absorption when resume() rejects, (d) resume() is NOT called when document.visibilityState is 'hidden' (defensive).
- **D-16:** **No App-level integration test required for audio resume.** The hook test is sufficient because the visibility-resume path is fully internal (D-08: hook surface unchanged → App.tsx is unchanged → no integration seam to exercise). Mirrors the Phase 5 split where the wake-lock unit tests in `useWakeLock.test.tsx` carry most of the coverage and the App.wakeLock.test.tsx tests cover only the request/release call-site integration. The audio resume in 5.1 has no equivalent App-level call sites.

### UAT
- **D-17:** **Manual UAT runs on real Safari for both fixes.**
  - Audio resume: real iPhone Xs Max iOS 18.7.8 Safari, 5+ min running session, lock 30 s, unlock, audio cues resume within ≤1 phase boundary. Append to `05.1-UAT.md` (or equivalent name per planner) with device + iOS version + observed behavior.
  - Orb visual: Safari desktop screenshot at peak inhale (orb at MAX_SCALE) showing inner orb meeting outer ring. Compare against the existing `05-04-UAT-LOG.md` Finding 1 screenshot baseline. Optional: capture iOS Safari at peak inhale too — same fix, same render path.
- **D-18:** **Phase 1–5 automated tests must all stay green.** Success-criterion 3 ("All Phase 1–5 automated tests remain green; no regressions in Chromium, Firefox, or jsdom test runs") is a goal-backward gate. The new audio test additions are net-add only; the orb CSS change is constrained by D-13 to not affect existing render paths.

### Claude's Discretion
The following implementation details are explicitly left to research/planning. The decisions above bound the design space; the planner picks within those bounds:
- Exact form of `audioEngine.resume()` — whether the method short-circuits on `closed === true` (probably yes, mirrors `scheduleLeadIn`'s `if (closed) return`) and whether it returns `Promise<void>` or `Promise<boolean>` (either acceptable; D-08 says no public hook return surface, and D-04 says caller doesn't branch on it — return type is for testing ergonomics).
- Whether the outer-ring CSS fix uses option (1) content-box width/height or option (2) negative inset. Both satisfy D-10. Planner picks based on which works cleaner across Safari + Chromium + Firefox in the test suite, with explicit cross-browser checks (the Plan 02-02 test pattern is precedent for cross-browser CSS verification).
- Whether the test file is `useAudioCues.resume.test.tsx` (new file) or appended to the existing `useAudioCues.test.tsx`. Same decision Phase 5 made for `useWakeLock.test.tsx` (single file). Either is fine; planner picks based on file size growth.
- Whether `console.debug` lines are added in the resume failure path. D-09 makes this optional.
- Whether the visibility listener uses `document.visibilityState === 'visible'` or `!document.hidden`. Both equivalent; planner picks for symmetry with `useWakeLock.ts:82`.

## Re-Plan Section — Plan 05.1-04 (post Plan 02 UAT FAIL, 2026-05-10)

Plan 02 (D-10/D-11/D-12/D-13) shipped and the 1.5px border-box compensation works on iOS Safari (PASS). Safari Desktop UAT (Task 3 of Plan 03) FAILED with a much larger gap. Web Inspector data at `data-progress="0.971"` (`transform: scale(0.988016)`):

| Element | getBoundingClientRect().width |
|--------|-------------------------------|
| `div[role="img"][data-phase]` (container, `width: var(--orb-size)`) | **360.00 px** ✓ |
| `.orb-ring--outer` (`style={{ inset: '-1.5px' }}`) | **363.00 px** ✓ (D-10 fix works) |
| `.orb` (`className="orb absolute inset-0 ..."`) | **209.00 px** ✗ — collapsed to ≈ MIN_SCALE × 360 = 0.58 × 360 = 208.8 |

The orb's underlying box is ~58% of parent on Safari Desktop, so `transform: scale(0.988)` of 209 px = ~206 px (visible orb), not the expected ~356 px. The 75 px-per-side visible gap is **not** the 1.5px D-10 was targeting — it is a deeper Safari Desktop grid+abs-pos sizing collapse already documented at `BreathingShape.tsx:82` (the WR-03 comment). The inner ring already dodges this by using explicit `left:50%; top:50%; width:58%; height:58%; -translate-{x,y}-1/2`. The orb and the outer-ring span still rely on `inset-0` / `inset: -1.5px` shorthand, which Safari Desktop does NOT honor for size on absolutely-positioned children of `display: grid; place-items: center` parents.

## Re-Plan Section — Plan 05.1-06 (iOS audio resume root-cause fix, 2026-05-10)

Plan 01 (D-01..D-09) shipped per spec but empirically failed on real iPhone iOS Safari (see `05.1-UAT.md` Task 2 — diagnostic findings appended 2026-05-10). Root cause is twofold:

1. **WebKit-only `'interrupted'` AudioContext state.** iOS Safari puts the AC into a non-spec `'interrupted'` state on display sleep / lock screen, not `'suspended'` as the MDN spec describes. Plan 01 D-04 read the spec literally; reality has an extra state.
2. **`audioCtx.resume()` invoked from a `visibilitychange` handler is not a user gesture on iOS Safari.** Promise rejects with `InvalidStateError "Failed to start the audio device"` even though the AC transitions to `'suspended'` (normally recoverable). Plan 01 D-09 silent-absorbed this rejection, so audio dies invisibly while visuals continue.

Diagnostic confirmed on real device with temporary Web Inspector instrumentation (reverted; not committed). See `05.1-UAT.md` Task 2 block for raw capture + 6 directional inputs.

Plan 06 is scoped as the empirical-evidence-driven follow-up. It carries forward all Plans 01-05 locked decisions (D-01..D-28) — Plan 01's listener wiring + ref-gate + engine.resume() method all stand; Plan 06 widens the recovery path and adds the user-gesture affordance that the visibility handler cannot supply.

### Anti-Pattern Acknowledgments (from .continue-here.md blocking section)

Plan 06 explicitly mitigates both blocking anti-patterns from the Phase 5.1 `.continue-here.md`:

- **"Treating jsdom green as Safari green"** — Plan 06 will include a real-device UAT task before sign-off, the same way Plan 03 Task 2 covers Plan 01's audio path. The post-fix UAT MUST be performed on real iPhone iOS Safari with explicit pass/fail capture for both the suspended-recoverable path AND the closed-AC reconstruction path.
- **"Spec'ing iOS Safari behavior from MDN/spec without device confirmation"** — Plan 06's spec is already grounded in device-confirmed data (the diagnostic findings above). Any further behavior the planner needs (e.g. how iOS handles `addEventListener('statechange')` during lock; whether `createAudioEngine()` from a gesture chain succeeds when invoked synchronously from a tap handler) must be verified by an inline diagnostic spike, not assumed from spec.

### Plan 06 Decisions

#### Recovery State Surface

- **D-29:** **An explicit user-tappable affordance is rendered when the audio path enters `needs-resume` state.** Audio path is `needs-resume` when (a) `visibilitychange → visible` fires AND `engineRef.current !== null` AND (b) the subsequent `engineRef.current.resume()` rejects (or `audioCtx.state` is `'interrupted'` after the call). The affordance is the user-gesture context the visibility handler cannot supply. Pure-silent absorption (Plan 01 D-09) is relaxed for this specific failure class.
- **D-30:** **Affordance auto-hides once the audio path leaves `needs-resume`.** Bound to the same flag: when `audioCtx.state` transitions to `'running'` (either via gesture-attached resume or engine reconstruction), the flag flips false and the affordance disappears. No additional animation required for v1 (the calm/meditative aesthetic favors instant transitions; a fade can be revisited if it feels abrupt during UAT).
- **D-31:** **`setMuted(false)` calls `engineRef.current.resume()` synchronously and clears the `needs-resume` flag.** The mute toggle click IS a user gesture — wiring this gives a cheap parallel recovery path. User who instinctively un-mutes after unlock recovers audio without seeing the affordance. Mute purity is intentionally compromised: mute-toggle in `needs-resume` state == resume affordance. Both controls do the right thing when tapped.

#### Affordance Placement And Form

- **D-32:** **The MuteToggle component morphs into the resume affordance during `needs-resume`.** Concretely: in `MuteToggle.tsx` (which currently renders a speaker-on / speaker-muted glyph + aria-label `"Mute"` / `"Unmute"`), add a third visual state driven by a new `needsResume` prop. When `needsResume === true`, the button renders a resume glyph (planner picks — e.g., play-triangle, refresh-arrow, or sound-with-spark) and aria-label `"Resume audio"`. The click handler (already routed through App.tsx's `persistedSetMuted(!audio.muted)` line 459) MUST be extended at the App.tsx level (not inside MuteToggle) to call `audio.resume()` first when `needsResume` is true. This keeps the component prop-driven and avoids putting recovery logic in the leaf component.
- **D-32a:** **Reduced-motion behavior:** the morph is a state swap (instant), not an animated transition. Honors `prefers-reduced-motion` automatically because no animation runs.
- **D-32b:** **Accessibility:** screen-reader users get a fresh aria-label on the state change. Planner SHOULD verify the button receives an `aria-live="polite"` announcement or equivalent so the change is audible; this MAY be deferred to UAT if it requires significant infrastructure. Hit-area floor stays 44×44 px per the Phase 2 D-15 contract — unchanged.

#### Engine Reconstruction Fallback

- **D-33:** **If the affordance tap's `resume()` rejects, immediately reconstruct the engine in the same synchronous gesture chain.** Implementation outline:
  1. Tap handler: `await audio.resume()` (gesture-attached — succeeds for `'interrupted' / 'suspended'` recoverable cases).
  2. If `audio.resume()` succeeds (audioCtx.state → `'running'`), clear flag, done.
  3. If `audio.resume()` rejects OR `audioCtx.state` still not `'running'` after await, call `audio.reconstruct()` (new method, see D-34) synchronously from the same handler — iOS treats this as the same gesture context.
  4. Reconstructed engine inherits muted state + (per Phase 3 dual-anchor) a re-anchored time origin. See D-35 below for anchor handling.
- **D-34:** **`useAudioCues` exposes a new `resume()` method to App.tsx, and the engine factory gains a `reconstruct()` capability.** Concretely:
  - `useAudioCues` adds a public `resume(): Promise<void>` method on the returned object. App.tsx calls it from the mute-toggle handler when `audioStatus === 'needs-resume'`. (Plan 01 D-08 "no new public surface" is explicitly relaxed for Plan 06 — the gesture-attached recovery path REQUIRES a caller seam at the user-gesture level.)
  - `useAudioCues` adds a public `audioStatus: 'ok' | 'needs-resume' | 'unavailable'` field (or equivalent boolean `needsResume`). Drives MuteToggle's `needsResume` prop.
  - Implementation: when the visibility-handler's `engine.resume()` rejects, set the state. When a successful resume / reconstruction lands, clear it.
  - The `audioEngine.ts` `resume()` method (Plan 01 D-07) stays; Plan 06 adds a separate internal mechanism (NOT a new public engine method) for reconstruction. The hook-level `useAudioCues.resume()` decides internally whether to call `engineRef.current.resume()` or close+recreate the engine.
- **D-35:** **Anchor handling on reconstruction differs from anchor handling on plain resume.** Plan 01 D-06 said "no re-anchor on resume" — correct because the AC clock continues from where it was on resume. On reconstruction, the new AC's `currentTime` starts at 0 — the dual-anchor (Phase 3 D-13/D-14) MUST be re-anchored. Specifically: when `useAudioCues` reconstructs the engine, it re-captures `audioCtx.now()` and notifies App.tsx (via a new optional callback prop OR by re-running the existing lead-in-completion code path — planner picks). The breathing plan's `audioAnchorRef` in App.tsx is reset to the new anchor; subsequent `scheduleNextCue` calls compute against the new origin. WITHOUT re-anchoring, all cues schedule against an audioTime that doesn't exist on the new AC and silently fail to fire.
- **D-35a:** **Lead-in is NOT replayed on reconstruction.** The user was already in mid-session; they don't need 3-2-1 again. The first scheduled cue after reconstruction is the next In or Out cue at the next phase boundary (consistent with D-05).
- **D-35b:** **Mute state survives reconstruction.** New engine receives `setMuted(currentMutedState)` synchronously after construction. Audio comes back at the user's chosen volume / muted-ness.

#### State Tracking Mechanism

- **D-36:** **State tracking uses `audioCtx.addEventListener('statechange', ...)` driven by a single hook-level listener.** The engine factory wires the listener internally and exposes state transitions through a callback supplied at construction time, OR (planner discretion) `useAudioCues` reads `audioCtx.state` after every operation and pushes the value into React state. Statechange listener is preferred (reactive, no polling) but requires the engine to accept an `onStateChange` callback in its constructor. Either is fine; planner picks based on test ergonomics. NOT acceptable: polling on rAF or interval — wastes battery and conflicts with the calm hands-off ethos.
- **D-37:** **`audioCtx.state === 'interrupted'` is treated symmetrically with `'suspended'` in any state-machine read.** Both are non-terminal recoverable states. The hook's `audioStatus` field is `'needs-resume'` when state is `'interrupted'` OR `'suspended'` AND a prior visibility-resume attempt failed. The `'closed'` state is terminal and surfaces as `'unavailable'` — App.tsx fallback (Phase 3 D-10 visuals-only) takes over.

#### Failure Posture (Relaxed D-09)

- **D-38:** **`InvalidStateError` from `audioCtx.resume()` is captured and used to drive `audioStatus`, NOT silent-absorbed.** Concretely: the engine's `resume()` method (Plan 01 D-07) keeps its try/catch but the error class is inspected. `InvalidStateError` → set `audioStatus = 'needs-resume'` (via the statechange listener or direct flag set). All other errors continue the silent-absorb posture (Plan 01 D-09). This is the MINIMAL relaxation of D-09 that lets Plan 06 work.
- **D-39:** **The visibility-resume path's optimistic call is preserved.** Plan 01 D-03's visibility-handler call to `engine.resume()` stays — sometimes resume actually succeeds without a gesture (e.g., headphone-in event, brief lock, certain iOS versions). When it succeeds, no affordance is needed. When it rejects, the affordance becomes the user's recovery path. The visibility-handler does NOT skip the call just because we expect it to fail.

#### Test Infrastructure

- **D-40:** **`vitest.setup.ts` FakeAudioContext is extended further:**
  - Add `'interrupted'` to the valid state values for the `state` getter.
  - Add `_simulateInterrupted()` (test-only, underscore-prefixed) that sets `state='interrupted'`.
  - Add `_simulateResumeReject(errorName)` (test-only) that makes the next `resume()` call reject with the named error (defaults to `InvalidStateError`).
  - Add `dispatchStateChange()` (test-only) that fires a `statechange` event to test the listener path.
  - Existing FakeAudioContext shape (Plan 01 D-14 additions) stays intact.
- **D-41:** **Hook tests cover the four-state matrix:**
  - State: `'running'` → visibility resume call succeeds → audioStatus stays `'ok'`.
  - State: `'suspended'` → visibility resume call succeeds (FakeAudioContext flips to running) → audioStatus stays `'ok'`.
  - State: `'interrupted'` → visibility resume call rejects with `InvalidStateError` → audioStatus → `'needs-resume'`.
  - State: `'closed'` → engine resume short-circuits → audioStatus → `'unavailable'`.
- **D-42:** **Integration tests at App.tsx level cover the affordance + mute-toggle dual path:**
  - When `audio.audioStatus === 'needs-resume'`, MuteToggle renders with `needsResume` prop and morphed aria-label.
  - Click on the morphed MuteToggle calls `audio.resume()` first, THEN `audio.setMuted(!audio.muted)`.
  - If `audio.resume()` succeeds, `audioStatus` clears, affordance reverts to normal mute glyph.
  - If `audio.resume()` rejects again (simulated via FakeAudioContext), App.tsx calls the engine reconstruction path (planner names the seam) and verifies new engine's `setMuted(currentMuted)` is invoked synchronously.
