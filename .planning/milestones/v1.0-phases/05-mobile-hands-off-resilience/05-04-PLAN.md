**Scenario 1 — iOS Safari 16.4+ : screen stays awake for 10 min (ROADMAP SC2)**
- Device: iPhone with iOS 16.4 or later.
- Browser: Safari (NOT a third-party browser shell).
- Steps:
  1. Open the app URL in Safari.
  2. Set duration to 10 minutes (default is 5; click Increase Duration once).
  3. Tap Start session.
  4. Place phone face-up on a flat surface, screen on, do NOT touch it.
  5. Confirm Settings → Display & Brightness → Auto-Lock is set to a value SHORTER than 10 minutes (e.g. 30 seconds or 1 minute) — this is what we're testing against.
  6. Wait the full 10 minutes.
- Expected: Screen does NOT auto-lock. Session shows "Session complete" at the 10-minute mark.
- Pass criteria: Screen stayed awake for the full 10 minutes WITHOUT manual touch.
- Common failure modes:
  - Screen locks at the OS auto-lock interval → Wake Lock not granted (check Safari version is ≥16.4; verify HTTPS).
  - "Session complete" never shown → unrelated session-timing bug; capture and report separately.

---

**Scenario 2 — Android Chrome : screen stays awake for 10 min (ROADMAP SC2)**
- Device: Android phone with Chrome 85+ (effectively any modern Android).
- Steps: Identical to Scenario 1, replacing Safari with Chrome.
- Note: Set Android Settings → Display → Screen timeout to 30 seconds before the test.
- Pass criteria: Screen stayed awake for full 10 minutes.

---

**Scenario 3 — Phone-lock / unlock mid-session re-acquire (D-03)**
- Device: iPhone OR Android (run on whichever is available; ideally both for parity).
- Steps:
  1. Start an open-ended session (Increase Duration past 60 → "Unlimited" / open-ended).
  2. After ~30 seconds of running, manually press the power button to lock the phone.
  3. Wait 30 seconds with the phone locked.
  4. Unlock the phone (Face ID / fingerprint / passcode).
  5. The app tab should still be foregrounded (because we just unlocked into it). If the OS dropped focus, swipe back to the browser tab.
  6. Watch the screen for 1 full minute after unlock without touching it.
- Expected: The screen does NOT auto-lock during that minute (Wake Lock re-acquired via `visibilitychange → visible`).
- Pass criteria: Screen stayed awake for the full minute after unlock.
- Common failure modes:
  - Screen auto-locks ~30s after unlock → re-acquire failed silently (D-09 absorption may have hidden a real error). Open Safari/Chrome DevTools BEFORE the test, watch console for any lifecycle messages, report findings.
  - Tab loses foreground on unlock and never regains → not a Wake Lock issue; document the OS behavior and skip.

---

**Scenario 4 — Desktop Firefox 126+ silent-fallback / supported-path (D-09)**
- Device: Any desktop with Firefox 126 or later.
- Steps:
  1. Open `localhost:5173` (or your dev server URL) in Firefox.
  2. Open DevTools → Console.
  3. Click Start session.
  4. Watch console output during the lead-in and first minute of running.
  5. Visually scan the page during running for any Wake Lock-related text, badge, banner, or icon.
- Expected:
  - No console errors. No `Uncaught (in promise)` warnings related to Wake Lock.
  - No user-visible UI references "wake lock" anywhere on the page (D-12 / D-10).
- Optional sub-step: In `about:config`, set `dom.screenwakelock.enabled=false`, restart Firefox, repeat — confirm SAME outcome (zero artifact). This exercises the D-09 silent-fallback path explicitly.
- Pass criteria: Zero console errors AND zero user-visible artifact.

---

**Scenario 5 — Visual sweep for Wake Lock UI absence (D-12)**
- Device: Any (run on desktop for fastest sweep).
- Steps:
  1. Run the app via `npm run dev`.
  2. Visit every visible state: idle (settings + Start), lead-in (3 → 2 → 1), running, modal-confirm End dialog, ResetStatsDialog (after at least one session is recorded), Session complete.
  3. In each state, visually scan for any element referencing "wake lock", "screen", "awake", "stay on", "screen lock", or any battery/screen icon.
  4. Optionally: `grep -r -i "wake.*lock\|stay.*awake" src/` (excluding `useWakeLock.ts` and test files) to confirm no copy-string was added to UI components.
- Expected: Zero matches. The phase boundary forbids any UI surface for v1.
- Pass criteria: No element referencing Wake Lock or screen-awake state in any app state.

---

**Scenario 6 — Battery-low / power-saver-on (acceptable to skip per VALIDATION)**
- Device: Phone with ability to run battery low.
- Steps:
  1. Drain phone battery to <20% (or enable Low Power Mode on iOS / Battery Saver on Android).
  2. Start a session.
  3. Observe whether Wake Lock is granted or rejected; if rejected, confirm no user-visible artifact.
- Expected: Either path is acceptable (D-09 silent absorption). Pass criteria: zero user-visible artifact regardless of grant/reject.
- **NOTE: Per `05-VALIDATION.md` Manual-Only Verifications: "Acceptable to leave unverified in v1 — hook treats revocation as 'lock revoked → silently absorbed' via the same code path as any other rejection. Trust unit test of rejection handler."** Run only if the tester has time and a depleted device.
