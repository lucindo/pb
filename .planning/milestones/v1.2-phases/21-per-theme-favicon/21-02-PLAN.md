<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| localStorage (persisted prefs) → inline pre-paint script | The inline script reads `hrv:state:v1` from localStorage and resolves a theme id; only the theme id influences favicon construction. |
| build-time hex map → runtime SVG data-URI | Both `useFavicon` and the inline script build the favicon from a closed set of 5 hardcoded hex constants — no user/network input reaches the SVG. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-21-04 | Tampering / Injection | `index.html` inline script building the `data:image/svg+xml,...` URI from a hex value | mitigate | The resolved theme id is validated against the literal allowlist `['light','dark','moss','slate','dusk']` (the existing script already does this) before indexing the hex map; the hex map holds only 6-digit literals. A tampered localStorage theme value falls through to the `light` default — no attacker-controlled string reaches the SVG fill. |
| T-21-05 | Tampering / Injection | `useFavicon` SVG data-URI construction | mitigate | `buildFaviconDataUri` accepts only `Exclude<ThemeId,'system'>` (a 5-value union); `loadPrefs()` coerces invalid stored themes to `DEFAULT_THEME` (Phase 14 D-17). No arbitrary string reaches the SVG template. |
| T-21-06 | Denial of service | malformed localStorage crashing the pre-paint script | accept | The inline script's existing single `try/catch` already falls back to a safe default on any parse failure; the favicon extension stays inside that same catch (extended to set the `light` favicon). Worst case is the default favicon — no crash, no blank page. |
| T-21-07 | Tampering | inline hex map drift from `theme.css` / `faviconPalette.ts` | mitigate | `favicon.sync.test.ts` (Task 2 enables the inline-map assertion) fails the per-commit green-gate if the three hex copies diverge. |
</threat_model>
