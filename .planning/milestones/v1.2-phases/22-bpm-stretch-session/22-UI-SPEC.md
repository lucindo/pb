### 2. Stretch Fields Block (conditional, Stretch mode only)

Rendered in place of the single `bpm` stepper when `settings.mode === 'stretch'` (D-06).
All fields reuse `SettingsStepper` â€” no new component.

Field list and formatting:

| Field | Label EN | Label PT-BR | Format | Options |
|-------|----------|-------------|--------|---------|
| `initialBpm` | `"Start BPM"` | `"BPM inicial"` | `"{value} BPM"` | `BPM_OPTIONS` (1â€“7, step 0.5) â€” minimum 1.5 (see below) |
| `targetBpm` | `"Target BPM"` | `"BPM alvo"` | `"{value} BPM"` | `BPM_OPTIONS` filtered to `< initialBpm` |
| `holdInitialSeconds` | `"Warm-up"` | `"Aquecimento"` | `"{value}s"` or `"None"` / `"Nenhum"` when 0 | `[0, 15, 30, 45, 60]` |
| `holdTargetSeconds` | `"Cool-down"` | `"Resfriamento"` | `"{value}s"`, `"None"` / `"Nenhum"` when 0, `"Open-ended"` / `"Aberto"` | `[0, 15, 30, 45, 60, 'open-ended']` |
| `rampDurationMinutes` | `"Ramp"` | `"Rampa"` | `"{value} min"` | `[5, 10, 15, 20, 25, 30, 35, 40, 45, 50, 55, 60]` |

**initialBpm constraint:** minimum option is 1.5 (not 1.0) to ensure `targetBpm` always has at least one valid option (1.0). This constraint is enforced at the options-array level, not via stepper disable.

**targetBpm edge:** when `initialBpm` reaches the lowest allowed value (1.5), `targetBpm` options array = `[1.0]`. Both stepper arrows become disabled (no movement possible). The output still renders `1.0 BPM`. This is acceptable â€” the user cannot go lower than 1.5 for `initialBpm`.

Placement order within the stretch block: `initialBpm` â†’ `targetBpm` â†’ `holdInitialSeconds` â†’ `rampDurationMinutes` â†’ `holdTargetSeconds`. Reads as "start BPM, target BPM, warm-up hold, ramp, cool-down hold" â€” top-to-bottom journey order.

### 3. Computed Total Readout (SettingsForm addition)

Rendered below the stretch fields block, above the ratio stepper (which remains visible in both modes).

- Element: `<p>` or `<output>` with `aria-live="polite"`

- Source: CONTEXT.md D-08

Format rule: `M:SS` â€” minutes (no leading zero) + colon + seconds (always 2 digits). Example: 25 minutes 30 seconds â†’ `"25:30"`. 20 minutes â†’ `"20:00"`.

### 4. Live BPM + Stage Label (SessionReadout addition)

## Interaction Contracts

### Mode Picker Interaction

- Stepping from Standard â†’ Stretch: allowed only when gate is clear (computed stretch total â‰¥ 15 min). The increase button is disabled (`disableIncrease`) when the gate fails.
- Stepping from Stretch â†’ Standard: always allowed (decrease button unaffected by gate).
- When gate blocks Stretch, the hint `<p>` renders immediately below the mode stepper: `"Needs a 15+ min session"`.
- Gate evaluation: computed from default stretch field values even when mode is Standard, so Stretch is reachable from first visit (defaults yield 20-minute total â€” RESEARCH.md Pitfall 5 resolution).

### targetBpm Constraint

- `targetBpm` options are filtered to `BPM_OPTIONS.filter(v => v < initialBpm)` (D-01).
- When user adjusts `initialBpm`, if current `targetBpm >= initialBpm`, auto-correct `targetBpm` to the highest valid option below `initialBpm`.
- The stepper renders with the corrected value immediately â€” no flash of invalid state.

### Computed Total Live Update

- The computed total readout updates on every change to `holdInitialSeconds`, `rampDurationMinutes`, or `holdTargetSeconds`.
- Uses `aria-live="polite"` so screen readers announce changes when focus is idle.

### In-Session Controls

- Stretch fields (mode picker, initialBpm, targetBpm, holdInitialSeconds, rampDurationMinutes, holdTargetSeconds) are hidden during `isRunning === true`, consistent with existing bpm stepper hide-during-running pattern.
- The duration stepper remains available during running for the extend-duration interaction (unchanged).
- Live BPM and stage label render in SessionReadout only during a running stretch session (when `frame.currentBpm !== undefined`).

### Reduced Motion

- The BPM chip and stage label are static text â€” no animation. Reduced-motion is not applicable to these elements.
- Existing orb and modal reduced-motion rules in `theme.css` are unchanged.
