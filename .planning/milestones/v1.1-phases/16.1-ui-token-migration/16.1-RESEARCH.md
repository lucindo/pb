# Phase 16.1: UI Token Migration — Research

**Researched:** 2026-05-13
**Domain:** Tailwind v4 design-token theming; class-only refactor of 16 production TSX components
**Confidence:** HIGH

## Summary

Phase 16 shipped the theme machinery (5 `--color-breathing-*` token palettes + a 6th `system` resolver) but only 1 of ~17 user-facing components (`StatsFooter`) actually consumes the tokens — the other 15 still reference hardcoded Tailwind palette utilities (`text-teal-800`, `bg-white`, `text-slate-900`, etc.) that were baked at compile time. Result: a theme swap rebinds the `ThemePicker` selected option and a handful of leaf elements (the orb gradient, modal backdrop, the focus ring, `StatsFooter` text, the `LearnDialog` link block) — but the primary action button, dialogs, stepper, anchors, and the unselected picker buttons all stay teal-on-white in every palette. Phase 16 verification noted this explicitly (`16-VERIFICATION.md` accepted-gap #2; `16-REVIEW.md` WR-01).

**Key technical insight:** Tailwind v4 with `@theme { --color-breathing-X: ... }` auto-generates utility classes (`bg-breathing-X`, `text-breathing-X`, `border-breathing-X`, `ring-breathing-X`) whose compiled CSS is `background-color: var(--color-breathing-X)`. That means **both** `bg-breathing-accent` (clean) and `bg-[var(--color-breathing-accent)]` (verbose arbitrary-value) re-bind across `[data-theme]:root` overrides — they are functionally equivalent. The migration can pick whichever form is more readable; the existing codebase (`ThemePicker.tsx:32`, `StatsFooter.tsx:35,54`, `LearnAnchor.tsx:26`) uses arbitrary-value form. Phase 16 PATTERNS.md "shared patterns" section explicitly documents the arbitrary-value approach as the codebase convention.

**Primary recommendation:** Token-only class substitution across 16 production `.tsx` files. Zero new tokens needed — every banned class has an obvious semantic mapping to an existing `--color-breathing-*` token. The destructive `bg-red-700`/`text-white` pair on the End-session/Reset confirm buttons should stay (red is intentionally NOT theme-shifted — danger semantics). Test files assert no banned classes, so no test edits required beyond test updates for the components themselves.

### Alternatives Considered

| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| Arbitrary-value form `bg-[var(--color-breathing-accent)]` | Tailwind v4 utility form `bg-breathing-accent` | Both compile to identical CSS (`background-color: var(--color-breathing-accent)`). Utility form is shorter and reads more naturally. Arbitrary-value form is what Phase 16 already uses in `ThemePicker.tsx:32`, `StatsFooter.tsx:35,54`, `LearnAnchor.tsx:26`. Recommendation: arbitrary-value to match existing code; consistency over micro-readability. [VERIFIED: tailwindcss.com/docs/theme] |
| Inline `style={{ background: 'var(--color-...)' }}` | Tailwind class | Inline styles bypass Tailwind's variant system (`hover:`, `focus-visible:`, `motion-reduce:`, `disabled:`), forcing manual event-handler color logic. Reject. The codebase uses inline `style={}` only when CSS-variable value is computed at runtime from React state (`BreathingShape.tsx:137-140` — the In/Out phase text color) — that's a different pattern not relevant here. |
| Adding a semantic indirection layer (e.g., `--color-button-primary-bg` aliasing `--color-breathing-accent`) | Flat token direct use | Phase 16 D-01 already rejected semantic indirection ("no `--bg-token` aliasing"). 16.1 inherits that decision. Adding indirection now defeats Phase 16 D-01 retroactively. |

### Pattern 1: Token Substitution via Arbitrary-Value Form

**What:** Replace literal Tailwind palette utilities with `*-[var(--color-breathing-*)]` arbitrary-value form.
**When to use:** Any colorful className referencing a Tailwind palette that has a semantic equivalent in the `--color-breathing-*` token set.
**Example (already in codebase — canonical):**

```typescript

<div className="mt-6 text-center text-sm leading-6 text-[var(--color-breathing-muted)]">
  // ...
  <button
    className="inline-flex min-h-[44px] min-w-[44px] items-center justify-center px-2 underline underline-offset-2 text-[var(--color-breathing-accent)] hover:text-[var(--color-breathing-accent)] focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-breathing-accent focus-visible:ring-offset-2"
  >
    Reset
  </button>
</div>
```

### Pattern 3: Conditional Token Selection (LearnAnchor / SettingsAnchor)

**What:** Disabled-vs-enabled branches mix a tokenized leg (already correct) with a hardcoded leg (the migration target).
**Example (current state — `LearnAnchor.tsx:25-27`):**

```typescript

  : 'border-teal-200 text-teal-800 hover:bg-teal-50 active:bg-teal-100'         // all 4 → token
```

**Pattern after migration (recommendation):**

```typescript

  ? 'cursor-not-allowed border-[var(--color-breathing-muted)] text-[var(--color-breathing-muted)]'

```

### Anti-Patterns to Avoid

- **Hand-rolling a new semantic token layer.** Phase 16 D-01 chose flat override; 16.1 inherits. Don't add `--color-button-primary-bg` aliases.
- **Mixing Tailwind utility form (`bg-breathing-accent`) and arbitrary-value form (`bg-[var(--color-breathing-accent)]`) in the same file.** Pick one per-file (or globally) — the codebase convention is arbitrary-value.
- **Theming destructive red.** `bg-red-700` / `text-white` on End-session and Reset-stats confirm buttons is intentional — red conveys danger semantics and should NOT rebind. The `text-white` on these is paired with `bg-red-*` and is a permanent neutral (white-on-red is the standard danger pairing).
- **Editing the selected-state ThemePicker classes** — they already use tokens correctly. Only unselected-state classes (line 33) are in scope.
- **Editing `src/styles/theme.css`** — Phase 16 closed it. The contrast guard must stay green from the same token values; the migration only changes consumers.
- **Sed-style blind find/replace** — semantics differ per usage site (see §3 mapping ambiguity callouts). Each file requires per-component judgment.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Theme-aware utility classes | Custom `[data-theme] .my-btn { ... }` CSS rules | Tailwind v4's `--color-*` → utility codegen (existing) | Already in place; adding a parallel system creates two sources of truth |
| Semantic token aliases | `--color-button-primary` → `--color-breathing-accent` | Direct `--color-breathing-accent` reference | Phase 16 D-01 explicitly rejected indirection |
| Runtime color computation | `style={{ background: getThemeColor('accent') }}` | CSS `var()` resolution | Browser-native, free, themable via cascade |
| Per-component "dark-mode" branch in JSX | `theme === 'dark' ? 'bg-slate-900' : 'bg-white'` | Single tokenized className that rebinds via cascade | The whole point of the Phase 16 token system |

**Key insight:** Phase 16 already built every piece of infrastructure 16.1 needs. The migration is class-string substitution — no logic, no new patterns, no new tests beyond updating existing class-assertion regexes (and the search above shows no test currently asserts a banned literal, so even that is empty work).

## Common Pitfalls

### Pitfall 1: Migrating destructive red

**What goes wrong:** Search-and-replace `bg-teal-700 text-white` → tokens accidentally catches `bg-red-700` adjacent `text-white` and turns the red destructive button into a theme-shifting accent button.
**Why it happens:** The two button shapes are visually similar (`min-h-12 rounded-full px-5 py-2 …`) and a grep-then-edit pass that's not narrow enough touches both.
**How to avoid:** Per-file edits with eyes on the danger semantic. The red destructive primary is `bg-red-700 hover:bg-red-800 active:bg-red-900` — never `bg-teal-*`. Use a targeted search like `bg-teal` (not `bg-(teal|red)`) when doing find-and-replace.
**Warning signs:** EndSessionDialog or ResetStatsDialog confirm button no longer renders red, or the shadow color drops the red tint.

### Pitfall 2: Breaking Dark-palette primary button readability

**What goes wrong:** Migrating `text-white` on `bg-teal-700` to `text-[var(--color-breathing-surface)]` is naively correct in Light/Moss/Slate (where surface = white) but produces near-zero contrast in Dark (where surface = `#1e293b` against the Dark `accent-strong = #5eead4` — both light colors).
**Why it happens:** The token system was designed by Phase 16 to make every color shift per palette; the implicit "white text on accent" pairing was never re-checked when Dark's accent flipped from teal-700 to teal-400.
**How to avoid:** Per-palette manual button-readability UAT before merging the SessionControls migration. Or: introduce a `--color-breathing-on-accent` token (Phase 16 D-02 scope expansion — escalate first). Or: keep `text-white` literally on the primary button as a permanent neutral, accepting that it might over-contrast in Light (white-on-grey-blue is fine, just less saturated). **Recommendation:** flag this as the single human-verify checkpoint of 16.1.
**Warning signs:** Start/Stop button text invisible or near-invisible in Dark theme; user reports "where did the button go?"

### Pitfall 3: `bg-white/70` alpha-modifier loses theming in 4 alpha sites

**What goes wrong:** `bg-white/70` matches the regex but if migrated to `bg-[var(--color-breathing-surface)]/70`, Tailwind v4 may or may not support the `/N` alpha modifier on `var()`-typed arbitrary values. If unsupported, the class no-ops and the element renders transparent.
**Why it happens:** Tailwind v4 added rich `color/N` alpha modifier support but the parser pathway for `[var(--x)]/N` is newer and not universally documented.
**How to avoid:** Test the migrated form in browser before committing. Fallback: use CSS-native `rgb(from var(--color-breathing-surface) r g b / 0.7)` inline `style={}` — works in all modern browsers (Chrome 119+, Safari 16.4+, Firefox 113+). Or: introduce a separate `--color-breathing-surface-70` token in `theme.css` (Phase 16 scope; flag).
**Warning signs:** Cards, fieldsets, time chips, anchor buttons render with no background fill after migration; backdrop blur is visible all the way through.

### Pitfall 4: Tailwind v4 utility form silently regresses

**What goes wrong:** Mixing `bg-breathing-accent` (utility form) with `bg-[var(--color-breathing-accent)]` (arbitrary-value form) in the codebase produces identical CSS — but a future code review may "clean up" one form to the other and accidentally drop the `breathing-` prefix (writing `bg-accent` thinking it's equivalent), which is undefined and renders as no fill.
**Why it happens:** Tailwind v4 auto-prefixed utility names (`bg-breathing-accent`) are easy to mistake for shorter names.
**How to avoid:** Pick one form per the codebase convention. The codebase uses arbitrary-value form (`bg-[var(--color-breathing-accent)]`) in Phase 16's hand-written code (`ThemePicker.tsx`, `StatsFooter.tsx`, `LearnAnchor.tsx`). Stay with arbitrary-value form for 16.1 — consistency makes future audits trivial (`grep -E 'bg-\\[var\\(--color-breathing' src` returns the full migration result).
**Warning signs:** Inconsistent class shapes across files; visual review shows some buttons re-tint per theme while sibling buttons don't.

### Pitfall 5: The Phase 16 ThemePicker unselected-state regression (the original sin)

**What goes wrong:** `ThemePicker.tsx:33` was the first new component built on top of Phase 16's token system and partially defeated D-02 — selected buttons re-tint per palette, but unselected buttons stay teal-on-white in every palette. Re-typing this pattern in a downstream picker phase would propagate the same regression.
**Why it happens:** Phase 16 reused the "secondary button" Tailwind class string from `ResetStatsDialog`/`SettingsStepper` (which was authored Phase 2-era, pre-token-system) without re-checking it against the new themable surface.
**How to avoid:** 16.1 fixes this for the existing pickers AND establishes the canonical tokenized "secondary button" class string. Future Phase 17/18/19 pickers should copy the tokenized form, not the Phase 2 form.
**Warning signs:** Dark/Dusk theme picker: selected button looks fine, unselected buttons still teal-on-white. (Pre-16.1: this IS the current state.)

## Open Questions

1. **Should the `text-white` on the primary action button (`bg-teal-700` → `bg-[var(--color-breathing-accent-strong)]`) become `text-[var(--color-breathing-surface)]`, stay as `text-white`, or escalate to a Phase 16 D-02 token surface expansion adding a new `--color-breathing-on-accent` token?**
   - What we know: `text-white` is a permanent neutral in the existing teal-on-white design; theme rebinding shifts the accent to varied hues; the cleanest theme-aware mapping is `text-[var(--color-breathing-surface)]` (works in 4 of 5 palettes where surface=white; Dark uses surface=#1e293b against accent-strong=#5eead4, which is *high* contrast but light-on-dark which is visually unconventional for primary CTAs).
   - What's unclear: the user/designer's intent — does the primary action button in Dark stay "light text on teal" (matching all other palettes) or invert to "dark text on light teal" (matching the cooler Dark scheme)?
   - Recommendation: lock in /gsd-discuss-phase as a discretion-or-decision: option (a) `text-[var(--color-breathing-surface)]` everywhere with a per-palette UAT, option (b) keep `text-white` as the permanent neutral on the primary, option (c) escalate and add `--color-breathing-on-accent` (out-of-scope token expansion).

2. **Should `bg-white/70`, `bg-white/80`, `border-white/80` alpha-modifier forms be migrated to `bg-[var(--color-breathing-surface)]/70` (assumes Tailwind v4 supports `/N` on arbitrary-value), or to a CSS-native `rgb(from ...)` form, or to a new dedicated `--color-breathing-surface-70` token?**
   - What we know: 5 sites use alpha-modifier forms; the simple-regex success criterion #2 catches them; the unmigrated form leaves card surfaces visually unthemable.
   - What's unclear: Tailwind v4's parser behavior for `bg-[var(...)]/N`.

   - What's unclear: scope creep vs. completeness.
