# Phase 16.1: UI Token Migration — Pattern Map

**Mapped:** 2026-05-13
**Migration surface:** 16 production `.tsx` files + `src/index.css` + 1 new token + 1 new guard test
**Reference sites (gold patterns):** 3 already-correct files (`ThemePicker.tsx` selected-state, `StatsFooter.tsx`, `LearnAnchor.tsx` disabled-leg)
**Match quality:** Migration is a refactor — every target has an in-codebase analog.

---

## How to read this map

Unlike a new-build phase, the migration surface IS the work. The value of this map is two-fold:

1. **Reference sites (gold patterns):** Files where the `var(--color-breathing-*)` recipe is already correct. The executor copies these verbatim.
2. **Migration target categorization:** Each of the 16 target files is grouped by *type of change* (substitution vs state-dependent vs alpha vs index.css vs dialog chrome), and pointed at its closest reference site.

The token-definition site (`src/styles/theme.css`) and contrast-test site (`src/styles/theme.contrast.test.ts`) are templates for D-01 (new `--color-breathing-on-accent` token) and D-04 (new `theme.no-hardcoded-classes.test.ts` guard) respectively.

**D-01 insertion rules (locked structural pattern — copy from theme.css verbatim):**

1. **Token name:** `--color-breathing-on-accent` (CONTEXT.md D-01 — symmetric with `--color-breathing-accent-strong`, NOT `on-accent-strong`).
2. **Position in each block:** immediately after `--color-breathing-muted` and before the orb-gradient comment block. One insert per palette × 5 palettes (Light/@theme + Dark + Moss + Slate + Dusk).
3. **Naming/spacing convention:** match the existing 2-space indent + ` --color-breathing-X: #hex;` shape verbatim. No alignment columns; existing tokens are not aligned.
4. **Value selection (planner's discretion, per CONTEXT.md "Claude's Discretion"):** pick a hex per palette so that `text-[var(--color-breathing-on-accent)]` on `bg-[var(--color-breathing-accent-strong)]` clears the THEME-05 ≥ 1.5 luminance contrast floor. Suggested seed values (planner refines):
   - Light: `#ffffff` (white on `#475569` slate-700 → very high contrast, definitely > 1.5)
   - Dark: `#0f172a` (slate-900 on `#5eead4` pale-teal → dark-on-light, clears Pitfall 2)
   - Moss: `#ffffff` (white on `#115e59` deep teal → very high contrast)
   - Slate: `#ffffff` (white on `#3730a3` indigo-700 → very high contrast)
   - Dusk: `#1e0550` (deep violet on `#f59e0b` amber → dark-on-warm, clears Pitfall 2)
5. **Do NOT touch:** orb gradient tokens, ring tokens, backdrop token, `--shadow-breathing-card`, the `.orb-*` rules, the reduced-motion `@media` block. None of these need to change for D-01.
6. **Contrast guard impact:** the existing 5-iteration THEME-05 test (`theme.contrast.test.ts`) does NOT test accent-strong vs on-accent today; CONTEXT.md "Specifics" notes "adding `on-accent` extends it to a 6th check (accent-strong vs on-accent)". The executor MAY extend the contrast test to add this 6th iteration (per palette), OR rely on the new `theme.no-hardcoded-classes.test.ts` only — planner's call (CONTEXT.md "Claude's Discretion"). **Recommended:** extend the contrast test by adding one more `it()` inside the same `describe.each` block — see §3 below for the exact pattern.

```typescript
// src/styles/theme.no-hardcoded-classes.test.ts
//
// Phase 16.1 THEME-UI-01 / D-04: regression guard — no production .tsx file
// references the 10 banned hardcoded Tailwind palette utilities. Long-term safety net
// for Phase 17/18/19 to prevent silent re-introduction of theme-blind colors.

/// <reference types="node" />

import { describe, it, expect } from 'vitest'
import { readFileSync, readdirSync, statSync } from 'node:fs'
import { resolve, join } from 'node:path'

// CONTEXT.md D-04: the 10 banned regex patterns (authoritative for both migration AND guard).
const BANNED_PATTERNS: ReadonlyArray<readonly [string, RegExp]> = [
  ['text-slate-N', /\btext-slate-[0-9]/],
  ['bg-slate-N', /\bbg-slate-[0-9]/],
  ['border-slate-N', /\bborder-slate-[0-9]/],
  ['text-teal-N', /\btext-teal-[0-9]/],
  ['bg-teal-N', /\bbg-teal-[0-9]/],
  ['border-teal-N', /\bborder-teal-[0-9]/],
  ['text-white', /\btext-white\b/],
  ['bg-white', /\bbg-white\b/],
  ['text-black', /\btext-black\b/],
  ['bg-black', /\bbg-black\b/],
]

// CONTEXT.md D-04: exclude *.test.tsx, *.stories.tsx, and src/themes/** (token defs).
function collectTsxFiles(dir: string, acc: string[] = []): string[] {
  for (const entry of readdirSync(dir)) {
    const full = join(dir, entry)
    const st = statSync(full)
    if (st.isDirectory()) {
      // Skip src/themes if/when it exists (CONTEXT.md D-04 exclusion).
      if (entry === 'themes') continue
      collectTsxFiles(full, acc)
    } else if (entry.endsWith('.tsx') && !entry.endsWith('.test.tsx') && !entry.endsWith('.stories.tsx')) {
      acc.push(full)
    }
  }
  return acc
}

const SRC_DIR = resolve(__dirname, '..')
const TSX_FILES = collectTsxFiles(SRC_DIR)

describe.each(BANNED_PATTERNS)('banned class %s', (name, pattern) => {
  it(`is not present in any production .tsx file`, () => {
    const hits: string[] = []
    for (const file of TSX_FILES) {
      const text = readFileSync(file, 'utf-8')
      if (pattern.test(text)) hits.push(file)
    }
    expect(hits, `Found banned ${name} in:\n${hits.join('\n')}`).toEqual([])
  })
})
```

**Structural mirrors (the executor MUST preserve these from `theme.contrast.test.ts`):**

1. Triple-slash `/// <reference types="node" />` directive — line 10 of `theme.contrast.test.ts`. Without it, `node:fs` types don't resolve.
2. `__dirname` + `resolve()` for path arithmetic — line 100 of `theme.contrast.test.ts`. Avoid hardcoded paths.
3. `describe.each` over an array of `[name, pattern]` pairs — mirrors line 128 of `theme.contrast.test.ts` iterating over `CONCRETE_THEMES`. One `describe` per banned pattern produces one named test per pattern; failure messages name the violator file. (Alternative: single test that asserts all patterns absent in one pass — fewer test results but worse failure diagnostics. **Recommended: per-pattern.**)
4. Zero new npm deps — uses only `node:fs`, `node:path` already proven in `theme.contrast.test.ts`.

Add a second `it()` block inside the same `describe.each(CONCRETE_THEMES)` (line 128 of `theme.contrast.test.ts`):

## 5. Anti-Patterns to Avoid

These are surfaced from `16-VERIFICATION.md` (Phase 16 verification), `16-REVIEW.md` (Phase 16 code review WR-01..WR-06), and the Phase 16 ThemePicker regression that triggered Phase 16.1's existence.

### AP-01 — The "Phase 16 original sin" (WR-01 — the bug 16.1 closes)

**What:** Mixing tokenized classes (selected state) with hardcoded teal/slate classes (unselected state) in the same conditional branch on the same element. ThemePicker.tsx:32-33 is the canonical example.

**Why it failed:** Selected button retints per palette; unselected buttons stay teal-on-white in every palette. Most jarring on Dark/Dusk where the selected button shifts to amber-on-violet / teal-on-dark while neighbors remain teal-on-white. Phase 16 verification flagged this as accepted-gap #2 ("Theme propagation gap — 58 hardcoded color-class lines across 16 components ... will be resolved by Phase 16.1").

**How to avoid in 16.1:** Every ternary state-branch in Group P-STATE must produce a className where BOTH branches use `var(--color-breathing-*)` tokens. The Reference site C migration shape above (`LearnAnchor.tsx`) is the canonical template; copy it verbatim.

**Sentinel:** After Group P-STATE merges, search the migration-surface files for `'border-teal-'`, `'bg-teal-'`, `'text-teal-'` string literals — should be zero matches outside the danger-pair on EndSession/ResetStats Confirm buttons.

### AP-02 — Migrating the danger pair (Pitfall 1)

**What:** Search-and-replace `bg-teal-700 text-white` → tokens accidentally catches `bg-red-700 text-white` in the same file.

**Why it fails:** Destructive End/Reset buttons lose the red danger semantic and rebind as theme-accent buttons. User no longer sees "this is destructive".

**How to avoid:** Per-file edits, eyes on the danger semantic. The danger-button pattern is `bg-red-700 hover:bg-red-800 active:bg-red-900 shadow-red-900/20`. Do NOT touch any line containing `red-`.

**Sentinel:** After Group P-DIALOG merges, `grep -E '\\bbg-red-' src/components/EndSessionDialog.tsx src/components/ResetStatsDialog.tsx` should still show the destructive buttons untouched (4 matches across 2 files).

### AP-03 — Dropping the Tailwind v4 prefix on `ring-breathing-accent` (Pitfall 4)

**What:** Mistaking `ring-breathing-accent` (Tailwind v4 auto-prefixed utility, references `--color-breathing-accent` automatically) for shorthand and "cleaning it up" to `ring-accent` (undefined; renders no ring).

**Why it fails:** Phase 16 accessibility tests (`App.dialog.test.tsx`, `App.settings.test.tsx`) assert `focus-visible:ring-breathing-accent` is present on every focusable button. Dropping the `breathing-` prefix breaks these tests AND removes the focus indicator from screen-reader users.

**How to avoid:** **DO NOT change** `focus-visible:ring-breathing-accent` anywhere. It is already token-aware via Tailwind v4's `@theme` codegen. D-06 (CONTEXT) explicitly locks accessibility-surface verbatim.

**Sentinel:** After every plan merge, run `grep -c 'focus-visible:ring-breathing-accent' src/components/*.tsx src/app/App.tsx` and confirm the count matches the pre-migration count.

### AP-04 — Inventing a parallel utility form (Pitfall 4, RESEARCH §"Anti-Patterns")

**What:** Mixing `bg-breathing-accent` (Tailwind v4 utility form — works because of `@theme` codegen) with `bg-[var(--color-breathing-accent)]` (arbitrary-value form — explicit `var()` reference).

**Why it fails:** Both produce identical CSS but the codebase convention is arbitrary-value (Reference sites A, B, C all use it). Mixing forms makes future grep audits brittle: `grep -E 'bg-\\[var\\(--color-breathing'` no longer enumerates the full migrated surface.

**How to avoid:** **Use arbitrary-value form exclusively.** CONTEXT.md D-05 locks this: "Follow the existing codebase convention ... Do NOT introduce a parallel pattern (`@theme` utility names like `bg-breathing-accent`)".

**Sentinel:** `grep -E '\\b(bg|text|border)-breathing-(bg|surface|accent|accent-strong|muted|on-accent|bg-soft|bg-edge)\\b' src/**/*.tsx` should match zero lines. (The `ring-breathing-accent` utility on focus rings is the explicit exception — see AP-03.)

### AP-05 — Sed-style blind find/replace (RESEARCH §"Anti-Patterns")

**What:** Running a single sed/regex pass across all 16 files swapping `text-teal-800` → `text-[var(--color-breathing-accent-strong)]` blindly.

**Why it fails:** Semantics differ per usage site. `text-teal-800` on a secondary button (Group P-STATE) maps to `accent-strong`; `text-teal-700` on the brand eyebrow (App.tsx:599) maps to `accent` (medium emphasis, RESEARCH §3 ambiguity #4). A blind pass collapses both into one mapping.

**How to avoid:** Per-file judgment. Use the RESEARCH §3 mapping table as the authoritative lookup, but check each occurrence's semantic role (label/body/button/border/anchor) before substituting.

**Sentinel:** None automated — this is the per-component-review discipline. The new `theme.no-hardcoded-classes.test.ts` only catches the *presence* of banned classes, not semantic mismatches in their replacements.

### AP-06 — Skipping the alpha-modifier probe (Pitfall 3)

**What:** Assuming Tailwind v4 supports `bg-[var(--color-breathing-surface)]/70` and migrating all 5 alpha sites in one pass without browser-rendering verification.

**Why it fails:** Tailwind v4's `/N` alpha-modifier parser pathway for `var()`-typed arbitrary values is newer and not universally documented (RESEARCH §"Pitfall 3", Assumption A1). If unsupported, all 5 alpha sites silently render with no background fill.

**How to avoid:** CONTEXT.md D-02 mandates the Wave 0 probe before committing the strategy. Run the probe on ONE file (e.g., LearnAnchor.tsx:24). If green → migrate the other 4 sites with the same form. If red → fall back to inline `style={{ backgroundColor: 'rgb(from var(--color-breathing-surface) r g b / 0.7)' }}` (CSS-native, Chrome 119+, Safari 16.4+, Firefox 113+).

**Sentinel:** Wave 0 probe checkpoint produces an artifact (test result + visual snapshot) before any Group P-ALPHA work begins.

### AP-07 — Touching `src/styles/theme.css` chrome tokens for retuning (CONTEXT "Out of Scope")

**What:** While in `theme.css` adding `--color-breathing-on-accent`, retuning Light's `--color-breathing-accent: #64748b` because it looks bland.

**Why it fails:** Aesthetic palette retune is **deferred to a separate todo** (`.planning/todos/pending/2026-05-13-themes-aesthetic-refresh.md`). Touching existing palette values invalidates the Phase 16 contrast guard's recorded ratios (light 1.78, dark 4.85, moss 1.94, slate 2.56, dusk 1.87) and may regress THEME-05. CONTEXT.md "Out of scope (explicit)" line lists "New palettes, palette re-tuning (other than the 5 new `on-accent` values)".

**How to avoid:** Touch **only** the 5 new `--color-breathing-on-accent` insertions in `theme.css`. Leave all 17 existing token values byte-identical. Run `npx vitest run src/styles/theme.contrast.test.ts` after the edit; expect 5/5 still pass with the same ratios.

**Sentinel:** `git diff src/styles/theme.css` after the D-01 plan should show only +5 lines (one per palette), zero modifications to existing lines.

### AP-08 — Editing `useThemeChoice.ts` to subscribe to events (WR-02 — out of 16.1 scope)

**What:** Phase 16 code review WR-02 notes `useThemeChoice` doesn't subscribe to `hrv:prefs-changed` or `storage` events. The migration executor may be tempted to "fix" this while editing TSX files that consume it.

**Why it fails:** CONTEXT.md "Out of scope" includes "Component refactors beyond class-string substitution" and D-06 locks "class composition changes only". WR-02 is a real bug but it's a different phase's work (carry-forward).

**How to avoid:** Touch only className strings + `src/styles/theme.css` (D-01 insertion) + `src/index.css` (D-03 migration) + the new guard test (D-04). Do NOT touch `useTheme.ts`, `useThemeChoice.ts`, or any hook logic.

**Sentinel:** `git diff src/hooks/` after Phase 16.1 completes should show zero changes (or only test-file changes if the executor adds new hook tests — also out of scope unless explicitly planned).
