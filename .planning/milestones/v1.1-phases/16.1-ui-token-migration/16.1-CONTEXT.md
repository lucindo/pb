# Phase 16.1: UI Token Migration — Context

<domain>
## Phase Boundary

**In scope:**
- Migrate all hardcoded Tailwind color classes (`text-slate-*`, `bg-slate-*`, `border-slate-*`, `text-teal-*`, `bg-teal-*`, `border-teal-*`, `text-white`, `bg-white`, `text-black`, `bg-black`) on `src/**/*.tsx` → `--color-breathing-*` tokens.
- Migrate the equivalent hardcoded references in `src/index.css` (raw text color + `::selection`).
- Extend Phase 16 token surface with one new token: `--color-breathing-on-accent` (per-palette tuned foreground for `accent-strong` backgrounds).
- Add a Vitest regression-guard test that greps `src/**/*.tsx` for the 10 banned class patterns.
- Preserve THEME-05 ≥ 1.5 luminance contrast guard across all 5 palettes (light, dark, moss, slate, dusk).

**Out of scope (explicit):**
- Red destructive primary buttons (`bg-red-700` + `text-white` on EndSession / Reset confirms) — `red-*` is danger semantic, intentionally palette-independent and NOT part of the banned-class grep.
- New palettes, palette re-tuning (other than the 5 new `on-accent` values).
- ThemePicker UX, `data-theme` plumbing, or any other Phase 16 contract — those are locked.
- New npm dependencies.
- Test fixtures / Storybook (none reference the banned classes per RESEARCH §2).
- Component refactors beyond class-string substitution.

</domain>

<decisions>
## Implementation Decisions

### D-01 — Add `--color-breathing-on-accent` token
Extend Phase 16 token surface with a new `--color-breathing-on-accent` CSS custom property, defined once per palette in the same `@theme` / `[data-theme="..."]` blocks that hold the existing 7 chrome tokens. Tune each palette's value so that `on-accent` over `accent-strong` clears the THEME-05 ≥ 1.5 luminance contrast floor. Used wherever the existing code has `text-white` sitting on an `accent-strong` background (primary action button, etc.). **Why this over keeping `text-white`:** Dark palette `accent-strong = #5eead4` is light; white-on-light regresses contrast. **Why this over mapping to `surface`:** Same problem in reverse on light palettes.

### D-02 — Wave 0 alpha-modifier probe before committing strategy
First plan in the phase MUST validate whether Tailwind v4 parses `bg-[var(--color-breathing-surface)]/70` (the arbitrary-value + alpha-modifier form). If it works, use it for all 5 alpha sites (`bg-white/70`, `bg-white/80`, `border-white/80`). If not, fall back to inline `style={{background:'rgb(from var(--color-breathing-surface) r g b / 0.7)'}}` for those sites. Decision locked into a follow-up plan by the probe outcome — planner produces one plan per outcome OR a single plan with both paths gated.

### D-03 — `src/index.css` in scope
Migrate `src/index.css` lines 6 and 22 (raw body text color + `::selection` background) to `--color-breathing-*` tokens in this phase. Goal language ("theme swap rebinds full interface") covers global stylesheet, even though success criterion #2 only audits `*.tsx`. Treat this as a small additional plan or fold into a chrome plan — planner's call.

### D-04 — Regression-guard Vitest test
Add `src/themes/__tests__/theme.no-hardcoded-classes.test.ts` (or sibling path matching the existing `theme.contrast.test.ts` location). Test reads every file under `src/**/*.tsx` and asserts none match the 10 banned regex patterns:
- `\btext-slate-[0-9]`
- `\bbg-slate-[0-9]`
- `\bborder-slate-[0-9]`
- `\btext-teal-[0-9]`
- `\bbg-teal-[0-9]`
- `\bborder-teal-[0-9]`
- `\btext-white\b`
- `\bbg-white\b`
- `\btext-black\b`
- `\bbg-black\b`

Test EXCLUDES `*.test.tsx`, `*.stories.tsx`, and `src/themes/**` (token definitions themselves). Test runs in the same Vitest suite as `theme.contrast.test.ts`. This is the long-term safety net for Phase 17/18/19 against silent regressions.

### D-05 — Migration target syntax: arbitrary-value `bg-[var(--color-breathing-*)]`
Follow the existing codebase convention (per RESEARCH §4 — already used in ThemePicker, StatsFooter, LearnAnchor). Do NOT introduce a parallel pattern (`@theme` utility names like `bg-breathing-accent`) in this phase even though both produce identical CSS in Tailwind v4. Consistency over micro-cleanup.

### D-06 — Migration is class-string substitution only
Class composition changes only. No component refactors, no prop renames, no behavior changes. Accessibility surface (focus-visible rings, hit-area floors, aria-*) preserved verbatim — success criterion #5.

</decisions>

<specifics>
## Specific Ideas

- The new token is **`--color-breathing-on-accent`** (not `on-accent-strong` — keep symmetric with existing `--color-breathing-accent-strong` parent).
- Banned-class regex above is authoritative for both the migration grep AND the new guard test — use the exact same patterns in both places.
- The Wave 0 probe (D-02) should be a *small* plan with a verifiable acceptance criterion: "render one component with `bg-[var(--color-breathing-surface)]/70` in a Vitest test, assert computed style `background-color` parses to expected rgba".
- Phase 16's RESEARCH already lists the 16 files + 58 occurrences — planner does not need to re-grep, but should cite line numbers from `16.1-RESEARCH.md` in `<read_first>` blocks.
- THEME-05 contrast guard is the existing 5-iteration test that checks each palette's `accent` vs `bg` luminance ratio. Adding `on-accent` extends it to a 6th check (accent-strong vs on-accent).

</specifics>

- Migration of `red-*` destructive utility classes — out of scope (danger semantic).
- Storybook fixture audit — none reference banned classes today (per RESEARCH §2); revisit if/when Storybook is added.
