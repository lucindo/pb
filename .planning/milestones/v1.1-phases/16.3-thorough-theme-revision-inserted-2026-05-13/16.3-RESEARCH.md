# Phase 16.3: Thorough Theme Revision — Research

**Researched:** 2026-05-13
**Domain:** CSS custom-property token system; open-source design-system palette extraction; per-palette theme curation
**Confidence:** HIGH

## Summary

Phase 16.3 replaces the chrome + orb gradient tokens in `src/styles/theme.css` for each of 5 palettes (Light, Dark, Moss, Slate, Dusk) with values **deliberately mapped from a named open-source design system** (e.g. Catppuccin Frappé → Dark) rather than picked ad-hoc from Tailwind's default palette. The work is pure stylesheet substitution against a frozen token surface defined in Phase 16 (D-14): per palette there are **8 chrome tokens + 4 orb gradient tokens + 2 orb-text tokens + 3 derived rgb(/alpha) tokens = 17 themable tokens** (verified by grep on theme.css). The token NAMES are locked from 16.1; only VALUES change. No `.tsx` files are touched (THEME-UI-01 invariant). No new npm dependencies are added. No new REQUIREMENTS.md IDs are added (B2 lesson from 16.2 — aesthetic redesign honors existing THEME-05 + THEME-UI-01 constraints).

The novel research dimension is the **token-mapping rubric** — a deterministic rule for routing source-palette role slots (e.g. Catppuccin's `base`/`mantle`/`surface0`/`text`/`peach`/`mauve`) onto OUR role slots (`--color-breathing-bg`/`-surface`/`-accent`/`-accent-strong`/`-muted`/`-on-accent` plus the 6 orb tokens). The rubric must preserve TWO invariants regardless of source convention: (1) `--color-breathing-accent-strong` vs `--color-breathing-on-accent` ≥ 1.5 WCAG luminance contrast (D-01 guard added in 16.1-01), and (2) `midpoint(orb-in-from, orb-in-to)` luminance > `midpoint(orb-out-from, orb-out-to)` luminance — i.e. **In is the light/airy pole** — and their ratio ≥ 1.5 (THEME-05/D-14). Some source palettes (notably the dark variants of Catppuccin/Rosé Pine/Tokyo Night/Nord) invert the In-as-light convention; the rubric must handle polarity-flip explicitly per palette.

Execution cadence is **B1 per-palette atomic-commit** (preserved from 16.2): each palette is its own task with its own contrast-guard verify and its own commit, so any regression bisects directly to the offending palette. The phase is **interactive** — each task starts by prompting the operator for a source design-system reference, then the agent extracts the source hex table (web-fetched + verified against the design system's canonical CSS/spec page), applies the mapping rubric, runs THEME-05 + accent-strong vs on-accent guards, and presents a per-palette UAT (smoke + visual integration + lead-in digit + active phase label legibility) BEFORE commit. The perceptual gate deferred from 16.2 is **satisfied here against curated palettes**, not ad-hoc retunes.

**Primary recommendation:** Plan as **6 plans, 5 waves**:
- **Plan 01 (Wave 0)** — Preflight: inventory the full token surface (17 themable tokens × 5 palettes = 85 declarations + 30 sentinel via `--color-orb-*`), establish a per-palette pre-edit contrast calculator (mirror 16.2's inline node-script pattern), confirm the mapping rubric against a worked example (e.g. Catppuccin Latte → Light). NO theme.css edits; output is a calculator script + a worked-example dry-run table in Plan 01 SUMMARY.
- **Plans 02–06 (Waves 1–5, one per palette)** — Interactive per-palette redesign. Each plan: (1) operator names the source design system; (2) agent extracts the canonical palette hex table; (3) agent applies the mapping rubric and writes the candidate token block; (4) automated guards run (THEME-05 + accent-strong vs on-accent + theme.no-hardcoded-classes + green-gate); (5) per-palette UAT in dev server; (6) atomic commit. Each plan is `autonomous: false` and contains an explicit `<read_first>` prompting operator input.
- Optional **Plan 07 (Wave 6)** — Phase close: STATE/ROADMAP/SUMMARY docs, ring-inner harmonization sweep (carry-forward from 16.2-01 deferred items), full-suite regression check.

### Pattern 2: Mapping rubric (NEW for 16.3)

**What:** Deterministic rule for routing source-palette role slots to OUR role slots. The rubric is what makes the work non-ad-hoc.

**OUR role slots** (10 themable + 3 derived rgb()/alpha + orb-text 2):

| OUR slot | Semantic role |
|----------|---------------|
| `--color-breathing-bg` | Page background (largest surface area) |
| `--color-breathing-bg-soft` | Card/secondary surface tint |
| `--color-breathing-bg-edge` | Edge/highlight tint (often near-white in Light, deeper in Dark) |
| `--color-breathing-surface` | Card/dialog inner surface |
| `--color-breathing-accent` | Secondary brand hue (e.g. picker hover, ring) |
| `--color-breathing-accent-strong` | Primary action background (Start/Stop button) |
| `--color-breathing-muted` | Secondary text + disabled foreground |
| `--color-breathing-on-accent` | Foreground that sits ON accent-strong (D-01 contrast pair) |
| `--color-orb-in-from` / `-in-to` | Orb gradient In phase (the "light/airy" pole) |
| `--color-orb-out-from` / `-out-to` | Orb gradient Out phase (the "dark/grounded" pole) |
| `--color-orb-in-text` / `-out-text` | Lead-in digit + active phase label color (renders on orb gradient at large size) |
| `--color-ring-outer` (rgb/alpha) | Outer reference ring — usually orb-in-from family at low alpha |
| `--color-ring-inner` (rgb/alpha) | Inner reference ring — usually orb-out-from family at low alpha |
| `--color-modal-backdrop` (rgb/alpha) | Dialog scrim — usually neutral dark at 0.30–0.50 alpha |

**Generic rubric** (apply per palette, refine when source has unusual slots):

| OUR slot | Rule | Notes |
|----------|------|-------|
| `bg` | Source's **base / mantle** (Light flavors) or **base / bg0** (Dark flavors). The widest-coverage surface tone. | Catppuccin Latte → `base #eff1f5`; Frappé → `base #303446`. |
| `bg-soft` | Source's **mantle / surface0** (one tier deeper than `bg` for Light, one tier lighter than `bg` for Dark)—the "card sits ON page" tone. | Catppuccin Latte → `mantle #e6e9ef`; Frappé → `surface0 #414559`. |
| `bg-edge` | Lightest available surface in Light flavors (`crust` inverted, or pure white); deepest available in Dark (`crust` direct). | This token is used at the *top-of-page eyebrow* — most invisible slot. |
| `surface` | Source's **surface0 / mantle / overlay0** — the dialog/card inner surface. Distinct enough from `bg` to read as "lifted". | In Catppuccin Latte: `surface0 #ccd0da`. In Mocha: `surface0 #313244`. |
| `accent` | Source's **primary semantic-blue accent** (Catppuccin `blue`, Nord `nord10`, Rosé Pine `foam`, Solarized `blue`). | The orb's "secondary" hue. |
| `accent-strong` | Source's **darker/saturated accent** for primary action — could be a darker tier of the same hue, or a contrasting accent (Catppuccin `mauve` or `sapphire`). MUST clear ≥ 1.5 vs `on-accent`. | This is the Start/Stop button bg. |
| `muted` | Source's **subtle / muted / overlay** — mid-luminance text + disabled foreground. | Catppuccin Frappé → `overlay1 #838ba7`. |
| `on-accent` | Source's **base** if `accent-strong` is dark, or `text` if `accent-strong` is light. **MUST be hue-paired with bg so the destructive End/Reset red+white combo still reads correctly** (existing D-01 invariant). | In Dark palettes typically = bg (e.g. Catppuccin Frappé `on-accent = base #303446`). |
| `orb-in-from`/`-in-to` | A **light-pole pair** in the source's blue/cyan/lavender family (Catppuccin `sky`/`sapphire`, Nord `nord7`/`nord8`, Rosé Pine `foam`). The from-stop and to-stop SHOULD be two tiers apart in luminance for gradient depth. | **In luminance MUST exceed Out luminance** (polarity invariant). |
| `orb-out-from`/`-out-to` | A **dark-pole pair** in a source's warm/saturated family (Catppuccin `peach`/`yellow`/`mauve`, Nord `nord12`/`nord14`, Rosé Pine `gold`/`rose`). | If source is a Dark variant where In and Out roles "feel inverted" (e.g. Mocha `surface0` darker than `peach`), invert the role-mapping not the gradient direction. |
| `orb-in-text` | Source's **darkest text / base** — must clear WCAG AA (4.5 body, 3.0 large) against `midpoint(orb-in-from, orb-in-to)`. | Lead-in digit renders at `text-7xl` (large-text per WCAG; threshold 3.0). |
| `orb-out-text` | Source's **darkest text** in the warm/Out family (e.g. brown/amber-deep) or `base` if `out` mid is light. | Active phase label renders at `text-5xl sm:text-6xl` (large-text per WCAG). |
| `ring-outer` (rgb/alpha) | `orb-in-from` channels at 0.5–0.7 alpha. | Pre-computed from the chosen In family. |
| `ring-inner` (rgb/alpha) | `orb-out-from` channels at 0.35–0.45 alpha. | Pre-computed from the chosen Out family. **NOTE:** 16.2-01 deferred ring-inner harmonization for Moss/Slate — 16.3 redesign of those palettes naturally cleans this up. |
| `modal-backdrop` (rgb/alpha) | Source's **crust / base** channels at 0.30–0.50 alpha (deeper for Dark palettes). | Locked formula — operator just confirms source. |

**Polarity invariant check** (after every rubric application, before contrast guard):
1. Compute `inMidLum = luminance(midpoint(orb-in-from, orb-in-to))`.
2. Compute `outMidLum = luminance(midpoint(orb-out-from, orb-out-to))`.
3. If `inMidLum < outMidLum` → **flip the In and Out role assignments** before writing the file. The source might be inverting the convention (e.g. some sources use "primary" for the brighter slot, others for the darker).

### Pattern 3: Pre-edit contrast calculator (inherited from 16.2-01)

**What:** An inline node script that mirrors `theme.contrast.test.ts`'s WCAG math — used to iterate hex candidates BEFORE applying file edits. Saves ~10 round-trips of edit → test → revert per palette.

**Example** (extrapolated from 16.2-01 SUMMARY §"Issues Encountered" — they used this same pattern):
```javascript
// Save as a scratch script; run with `node script.js`.
const srgbToLinear = c => c <= 0.03928 ? c / 12.92 : Math.pow((c + 0.055) / 1.055, 2.4)
const relLum = (r, g, b) => 0.2126 * srgbToLinear(r/255) + 0.7152 * srgbToLinear(g/255) + 0.0722 * srgbToLinear(b/255)
const hexToRgb = h => [parseInt(h.slice(1,3),16), parseInt(h.slice(3,5),16), parseInt(h.slice(5,7),16)]
const mid = (a, b) => [Math.round((a[0]+b[0])/2), Math.round((a[1]+b[1])/2), Math.round((a[2]+b[2])/2)]
const ratio = (a, b) => { const la=relLum(...a), lb=relLum(...b); return (Math.max(la,lb)+0.05)/(Math.min(la,lb)+0.05) }

// Iterate a candidate Light palette from Catppuccin Latte
const inFrom = hexToRgb('#dce0e8') // crust as orb-in-from (light pole)
const inTo = hexToRgb('#eff1f5')   // base as orb-in-to
const outFrom = hexToRgb('#fe640b') // peach as orb-out-from
const outTo = hexToRgb('#df8e1d')   // yellow as orb-out-to
console.log('In mid lum:', relLum(...mid(inFrom, inTo)).toFixed(4))
console.log('Out mid lum:', relLum(...mid(outFrom, outTo)).toFixed(4))
console.log('Ratio:', ratio(mid(inFrom, inTo), mid(outFrom, outTo)).toFixed(3))
// → check ≥ 1.5; check accent-strong vs on-accent same way
```

### Anti-Patterns to Avoid

- **Ad-hoc Tailwind-default picking** — That was 16.1/16.2's approach and is explicitly superseded by the open-source-source mandate.
- **Multi-palette commits** — Violates B1 cadence. Each palette is its own commit.
- **Adding new tokens** — Token NAMES are locked from 16.1 (D-01 + Phase 16 D-14). 16.3 only changes VALUES.
- **Touching `.tsx` files** — Violates THEME-UI-01 hardcoded-class guard contract. The guard is green at start and must stay green at every commit boundary.
- **Adding REQUIREMENTS.md entries** — 16.2 B2 lesson: aesthetic redesign has no new requirement IDs. Honors existing THEME-05 + THEME-UI-01.
- **Inverting In/Out polarity** — Some source palettes might suggest a dark In + light Out (e.g. Catppuccin Mocha's `surface0` is darker than `peach`). Always preserve `In = light pole, Out = dark pole` per the polarity-invariant check in the rubric.
- **Forgetting derived rgb()/alpha tokens** — `--color-ring-outer`, `--color-ring-inner`, `--color-modal-backdrop` use `rgb(R G B / alpha)` form, NOT hex. Update their channel triplets when the orb-in-from / orb-out-from / bg families shift, otherwise the ring/scrim hues will look "stuck on old palette" (this is exactly the Moss + Slate ring-inner issue 16.2-01 deferred).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| WCAG luminance / contrast math | Custom srgb→linear converter | `theme.contrast.test.ts` already has WCAG-canonical `srgbToLinear` / `relativeLuminance` / `contrastRatio` helpers. Mirror them in the pre-edit calculator script. | These are inline helpers — exact same math the guard runs. No risk of guard drift. |
| Hex parsing | Custom regex | `parseColorToRgb` in `theme.contrast.test.ts` handles `#rgb`, `#rrggbb`, `rgb(r,g,b)`. | If a future palette uses `oklch()` or `hsl()`, **add a parser branch to `parseColorToRgb` FIRST** — currently it throws. See Pitfall 5. |
| Per-palette UAT structure | Custom verdict format | 16.2-SUMMARY.md's 5-row table — Palette / data-theme attr / Visual integration verdict / Lead-in digit legibility / Active phase label legibility / Notes. | Pattern is locked in from 16.2-02. |
| Source-palette role-mapping | Ad-hoc Tailwind hue-eyeballing | Mapping rubric in this RESEARCH §"Pattern 2". | Rubric is what makes the work non-ad-hoc per phase goal. |
| Color extraction from web docs | Hand-typing hex tables | Use `gh api repos/{owner}/{repo}/contents/{path}` or raw.githubusercontent.com URLs to fetch canonical CSS/spec files. | Web-rendered hex tables are sometimes truncated by WebFetch summarization (Rosé Pine site rendered swatches-only — verified during research). |

**Key insight:** The guards are fully built; the only NEW work is the per-palette mapping rubric application + UAT. Resist any temptation to extend the contrast test, add new tokens, or refactor the orb-text consumer path — all are out of scope.

## Common Pitfalls

### Pitfall 1: Polarity inversion when sourcing from dark design systems

**What goes wrong:** Source palettes that target Dark mode (e.g. Catppuccin Mocha, Rosé Pine Main, Nord Polar Night) often have their "primary" or "blue/cyan" hues at *lower* luminance than the warm accent slots. If a naive rubric maps `source.blue → orb-in-from` and `source.peach → orb-out-from`, the In midpoint can end up DARKER than Out — violating the In-as-light-pole rule and likely failing the THEME-05 floor.

**Why it happens:** Open-source palettes are designed for **text-on-bg legibility**, not orb-gradient phase polarity. Their hue names (blue, peach, mauve) are descriptive of hue, not of luminance ordering.

**How to avoid:** ALWAYS run the polarity-invariant check after rubric application (compute `inMidLum` vs `outMidLum`, swap roles if `inMidLum < outMidLum`). The pre-edit calculator script (Pattern 3) handles this in 3 lines.

**Warning signs:** Contrast guard `theme=X` iteration fails with ratio < 1 (means In midpoint IS the darker pole — gradient is inverted).

### Pitfall 2: Derived rgb()/alpha tokens drift out of family

**What goes wrong:** `--color-ring-inner` / `--color-ring-outer` / `--color-modal-backdrop` are NOT hex — they use `rgb(R G B / alpha)` form where R/G/B are decimal channels of a *hex value from elsewhere in the palette*. If the rubric updates `orb-out-from` but leaves `--color-ring-inner` channels stale, the reference ring will render a foreign hue (this is exactly 16.2-01's Moss + Slate deferred issue).

**Why it happens:** The mapping rubric naturally focuses on the 14 themable tokens; the 3 derived ones are easy to forget.

**How to avoid:** Treat the 3 derived tokens as **rubric outputs**, not stand-alone slots. Recompute them from the chosen `orb-in-from` / `orb-out-from` / bg families:
- `ring-outer = rgb(hexToChannels(orb-in-from)) / 0.5..0.7`
- `ring-inner = rgb(hexToChannels(orb-out-from)) / 0.35..0.45`
- `modal-backdrop = rgb(hexToChannels(deepest-dark-tone)) / 0.30..0.50`

**Warning signs:** During per-palette UAT, the inner ring's color "doesn't belong" against the new orb-out gradient.

### Pitfall 3: D-01 accent-strong vs on-accent contrast regression

**What goes wrong:** The `accent-strong` (primary action button bg) and `on-accent` (foreground on it) pair must clear ≥ 1.5 luminance ratio per the 6th `it()` block added in 16.1-01. Source palettes sometimes pair a *muted* accent-strong with a same-family text color that doesn't clear the floor (e.g. Catppuccin Mocha `mauve #cba6f7` paired with `text #cdd6f4` is too low contrast).

**Why it happens:** Open-source palettes are optimized for syntax-highlighting contrast (color-on-bg), not for solid-button text legibility (text-on-color).

**How to avoid:** After mapping, run the D-01 check in the pre-edit calculator script (same WCAG math, different token pair):
```javascript
const ratio = ratio(hexToRgb(accentStrong), hexToRgb(onAccent))
console.log('accent-strong vs on-accent:', ratio.toFixed(3))
// MUST be >= 1.5
```
If it fails, swap `on-accent` to the deepest available source tone (Light flavors → use source's `text`; Dark flavors → use source's `base` or `crust`).

**Warning signs:** `theme.contrast.test.ts` shows the second `it()` block failing for a palette while the first passes (or vice versa).

### Pitfall 4: Gruvbox license metadata returns null

**What goes wrong:** `gh api repos/morhetz/gruvbox --jq '.license'` returns `None` even though the README explicitly states "MIT/X11". This is because Gruvbox doesn't have a top-level LICENSE file; the license is declared inline in the README only.

**Why it happens:** GitHub's license metadata auto-detection requires a LICENSE / LICENSE.md / COPYING file at the repo root.

**How to avoid:** Don't refuse a source on null license metadata. Check the README and the in-file copyright header (e.g. `colors/gruvbox.vim` has its own copyright statement). Gruvbox IS MIT-licensed per its own documentation.

**Warning signs:** Any tooling that auto-checks license metadata flags Gruvbox as "unlicensed."

### Pitfall 5: `parseColorToRgb` throws on `oklch()` and `hsl()`

**What goes wrong:** Some modern design systems (Radix Colors Pn3, Linear, Vercel internal) publish in OKLCH or HSL — not hex. If a per-palette plan tries to use one of these as-is, `parseColorToRgb` in `theme.contrast.test.ts` throws (line 64: `'Cannot parse color value: ...'. If a future palette uses oklch() or hsl(), add a parser branch here.'`).

**Why it happens:** The Phase 16 D-15 / RESEARCH explicitly scoped the parser to hex-only; adding OKLCH/HSL support was out of scope.

**How to avoid:** EITHER (a) only use hex-publishing sources for 16.3 (Catppuccin, Nord, Rosé Pine, Solarized, Gruvbox, Tokyo Night all qualify), OR (b) if operator names an OKLCH source, do the OKLCH→hex conversion in the mapping step BEFORE writing tokens to theme.css (Tailwind v4 supports OKLCH but our test harness doesn't). Recommend (a).

**Warning signs:** Contrast guard throws with the literal error string `Cannot parse color value: 'oklch(...)'`.

### Pitfall 6: Source-system attribution gating on copying hex values

**What goes wrong:** Some MIT licenses include an "include this notice in all substantial portions of the Software" clause. Worry that copying hex tables triggers this.

**Why it happens:** Misreading copyright law re facts vs creative expression.

**How to avoid:** Color hex values are facts (R/G/B integer triplets), not copyrightable creative expression. MIT's attribution clause applies to *substantial portions of the Software* — code, not numeric facts. Adding a one-line attribution comment to each palette block in theme.css (e.g. `/* Sourced from Catppuccin Latte (catppuccin/catppuccin MIT) */`) is best-practice goodwill, not a legal requirement. **[ASSUMED A1]** that this interpretation is correct; if operator wants a stricter interpretation, the attribution comment is cheap to add.

**Warning signs:** None — this is a research-time concern, not a runtime/test failure.

### Pitfall 7: Interactive plans + autonomous executors

**What goes wrong:** GSD plan executors default to `autonomous: true`. A per-palette plan that NEEDS operator input will deadlock at the prompt if the executor blindly proceeds.

**Why it happens:** Plan frontmatter `autonomous: true` (the default) instructs the executor to make all decisions itself. For 16.3 the operator MUST name the source per palette.

**How to avoid:** Each per-palette plan MUST set `autonomous: false` in frontmatter AND structure the first task's `<read_first>` block to PROMPT the operator explicitly. The plan-checker should flag any 16.3-NN plan with `autonomous: true` for palettes 02-06.

**Warning signs:** A per-palette plan executes without ever surfacing a "what design system?" prompt; commits a palette the operator didn't choose.

### Pitfall 8: WebFetch summarizes hex tables (rendered swatches lost)

**What goes wrong:** Web pages that render palettes as visual swatches (rosepinetheme.com/palette, gruvbox demos) — WebFetch's HTML→markdown conversion strips the swatch images and returns descriptions like "displays color swatches visually" rather than hex values.

**Why it happens:** WebFetch processes via a fast small model that drops non-text visual content.

**How to avoid:** When a source design system renders palettes as swatches in HTML, go directly to the raw CSS/JSON/spec file in the GitHub repo via `gh api repos/.../contents/...` or `raw.githubusercontent.com/...`. Verified during research: Rosé Pine's swatch-page returned no hex values, but `raw.githubusercontent.com/rose-pine/rose-pine-palette/main/dist/css/rose-pine.css` returned all 3 variants × 15 colors cleanly.

**Warning signs:** WebFetch result for a palette source contains the phrase "displays visually" or "swatches" without hex codes.

### Pre-edit contrast calculator script (recommended Wave 0 artifact)

**Source:** Pattern extrapolated from 16.2-01 SUMMARY §"Issues Encountered" + WCAG canonical math at `src/styles/theme.contrast.test.ts:20-90`.

```javascript
// .planning/phases/16.3-thorough-theme-revision-inserted-2026-05-13/scripts/contrast-calc.mjs
// (Optional Wave 0 artifact; alternatively run inline via `node -e`.)
// Mirrors theme.contrast.test.ts WCAG math exactly so guards never disagree with the pre-edit calc.

const srgbToLinear = (c) =>
  c <= 0.03928 ? c / 12.92 : Math.pow((c + 0.055) / 1.055, 2.4)

const relLum = (r, g, b) =>
  0.2126 * srgbToLinear(r / 255) +
  0.7152 * srgbToLinear(g / 255) +
  0.0722 * srgbToLinear(b / 255)

const hexToRgb = (h) => [
  parseInt(h.slice(1, 3), 16),
  parseInt(h.slice(3, 5), 16),
  parseInt(h.slice(5, 7), 16),
]

const mid = (a, b) => [
  Math.round((a[0] + b[0]) / 2),
  Math.round((a[1] + b[1]) / 2),
  Math.round((a[2] + b[2]) / 2),
]

const ratio = (a, b) => {
  const la = relLum(...a), lb = relLum(...b)
  return (Math.max(la, lb) + 0.05) / (Math.min(la, lb) + 0.05)
}

// Worked example: map Catppuccin Latte → Light
const palette = {
  bg:            '#eff1f5', // Latte base
  bgSoft:        '#e6e9ef', // Latte mantle
  bgEdge:        '#dce0e8', // Latte crust
  surface:       '#ccd0da', // Latte surface0
  accent:        '#1e66f5', // Latte blue
  accentStrong:  '#7287fd', // Latte lavender (deeper-feeling action)
  muted:         '#6c6f85', // Latte subtext0
  onAccent:      '#eff1f5', // Latte base (light text on dark accent-strong)
  orbInFrom:     '#1e66f5', // Latte blue (cool pole) — WAIT, will fail polarity if In ≠ light
  orbInTo:       '#04a5e5', // Latte sky
  orbOutFrom:    '#fe640b', // Latte peach (warm pole)
  orbOutTo:      '#df8e1d', // Latte yellow
  orbInText:     '#4c4f69', // Latte text
  orbOutText:    '#4c4f69', // Latte text
}

const inMid = mid(hexToRgb(palette.orbInFrom), hexToRgb(palette.orbInTo))
const outMid = mid(hexToRgb(palette.orbOutFrom), hexToRgb(palette.orbOutTo))
console.log('In mid lum:', relLum(...inMid).toFixed(4))   // 0.1893 — too dark!
console.log('Out mid lum:', relLum(...outMid).toFixed(4)) // 0.3717
// Polarity fails: in < out. Rubric must invert OR pick lighter In family.
// Real Latte → Light mapping needs orb-in-from = a light surface (e.g. crust + base mixed)
// and orb-out-from = a saturated mid-tone — iterate until polarity passes.
console.log('Ratio:', ratio(inMid, outMid).toFixed(3))    // would be < 1 — fail

console.log('accent-strong vs on-accent:',
  ratio(hexToRgb(palette.accentStrong), hexToRgb(palette.onAccent)).toFixed(3))
// 1.81 — passes D-01 floor
```
