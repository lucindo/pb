<objective>
Land the two cross-cutting "first paint = correct paint" pieces:

1. **`index.html` FOUC-prevention inline `<script>`** in `<head>` (S-03 locked shape). Reads `localStorage['hrv:state:v1']`, parses prefs.theme, resolves `'system'` (and any error path: missing key, parse failure, invalid theme id, matchMedia absent) via `matchMedia('(prefers-color-scheme: dark)')` with `'light'` as the catch-all fallback. Writes `<html data-theme='light'|'dark'|'moss'|'slate'|'dusk'>` synchronously before `<body>` is parsed.

2. **`src/storage/storage.ts` SYNC comment** (S-02). Add `// SYNC WITH index.html FOUC SCRIPT` immediately above the `STATE_KEY` export, documenting that the literal storage key string `'hrv:state:v1'` is hardcoded in two places (the TS export AND the inline HTML script) and any future `:v1 → :v2` bump must update both.

Purpose: THEME-04 specifically demands "applied before first paint." Without the FOUC script, a user with `theme: 'dark'` stored would see a flash of the default Light palette during the React hydration window (~50-200ms depending on bundle size and device). The script eliminates that flash. The SYNC comment makes the cross-file coupling discoverable so a future STATE_KEY bump doesn't silently break theme application on first paint.

Output: 1 edited HTML file, 1 edited TS file. Zero new files. Zero new dependencies (D-18).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → localStorage | The FOUC script reads localStorage which is same-origin only — cross-origin attackers cannot write to this app's localStorage. |
| localStorage → DOM | The parsed `prefs.theme` value flows from disk into `<html data-theme=>`. The script validates against a hardcoded allowlist `['light','dark','moss','slate','dusk']` before writing. |
| FOUC script → page | The script writes one DOM attribute and otherwise has no side effects. It does NOT call eval(), fetch(), or any sink that could be hijacked by a tampered storage value. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-03-01 | Tampering | localStorage payload (any same-origin script could pre-seed `prefs.theme = '<script>'` to attempt attribute injection) | mitigate | The FOUC script validates `t` against a hardcoded allowlist `['light','dark','moss','slate','dusk']`. Values failing the check are replaced by matchMedia resolution → `'light'` or `'dark'`. Even a malicious payload like `t = '<script>alert(1)</script>'` would fail `.indexOf(t) < 0`, be replaced with the matchMedia result, and `setAttribute('data-theme', 'light' | 'dark')` would write a safe string. No XSS surface created. |
| T-16-03-02 | Information disclosure | FOUC script reads localStorage | accept | The data read (`prefs.theme`) is non-sensitive (a UI enum). The script does not expose the read value anywhere — no console.log, no postMessage, no fetch. |
| T-16-03-03 | Denial of service | FOUC script throws (malformed JSON, missing key, matchMedia absent) | mitigate | All paths wrapped in `try/catch(_) { setAttribute('data-theme', 'light') }`. The catch block always succeeds because `setAttribute` is a synchronous DOM call with no failure modes. Worst case: the user sees the Light palette on first paint, which is the documented fallback per S-03. |
| T-16-03-04 | Tampering | Vite HTML transformation could strip the inline script | accept | RESEARCH §"Pattern 1" Vite gotchas: inline scripts without `type="module"` are preserved verbatim. The acceptance gate runs `npm run build` and inspects the built dist/index.html (see Task 1 verify). |
| T-16-03-05 | Repudiation | First-paint theme application is non-auditable | accept | UI preference application is not security-relevant; no audit trail needed. |
| T-16-03-06 | Elevation of privilege | (no privileged actions in this plan) | accept | The FOUC script writes one HTML attribute. No privileged APIs touched. |
| T-16-03-07 | Spoofing | Cross-tab script writes a malicious `prefs.theme` to localStorage before reload | mitigate | Same mitigation as T-16-03-01 — the allowlist filter applies regardless of write origin. Cross-origin tabs cannot write to this origin's localStorage (same-origin policy). |
</threat_model>
