<objective>
Create the two-hook orchestration layer that wires user theme choices to the cascading DOM attribute that the CSS plan 01 surface consumes:

1. **`useTheme()`** — App-side orchestrator (A-01/A-02). Owns React state seeded from `loadPrefs().theme`. Writes `document.documentElement.dataset.theme` via apply effect (resolving `'system'` via `matchMedia` per S-01). Attaches gated matchMedia listener (S-04: only when `state === 'system'`). Attaches cross-tab `'storage'` listener (A-04). Attaches same-tab `'hrv:prefs-changed'` CustomEvent listener (A-03). Returns `{ theme, setTheme }` (App ignores the return; the hook is called for its side effects).

2. **`useThemeChoice()`** — Picker-side state hook called from `ThemePicker.tsx` (deferred to plan 04). Mirrors `theme` locally for optimistic-UI rendering. Exposes `setTheme(next)` which writes via `savePrefs({...loadPrefs(), theme: next})` then dispatches `'hrv:prefs-changed'` with `detail: { key: 'theme', value: next }` (RESEARCH §"Pattern 3" generic shape for Phase 17/18/19 forward-compat).

Purpose:
- THEME-01 (user choice flows from picker click → persisted preference) — `useThemeChoice` provides the setter.
- THEME-02 (`'system'` follows OS via matchMedia) — `useTheme`'s gated mql lifecycle delivers live OS dark/light flips.
- THEME-04 (persists across reload, cross-tab + same-tab sync) — `useTheme`'s storage + custom-event listeners close the same-tab gap that the native `'storage'` event leaves open (RESEARCH Pitfall 4).

Output: 4 new files. Zero edits to any other file (D-21).
</objective>
