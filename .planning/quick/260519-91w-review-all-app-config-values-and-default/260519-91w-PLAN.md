In `src/storage/practices.ts` `coerceNaviKriyaSettings` (line ~66): after the
existing multiple-of-4 floor that produces `frontCount`, snap that value to the
nearest member of `NK_FRONT_COUNT_OPTIONS` (imported from
`src/domain/naviKriyaSettings.ts`). Ties round UP. Values below the new minimum
(100) snap up to 100. The returned `frontCount` MUST always be a member of
`NK_FRONT_COUNT_OPTIONS`.

Rationale (do not skip — this is the operator's locked decision): `SettingsStepper`
computes `selectedIndex = options.indexOf(value)`. A value not in `options` yields
index -1, which disables BOTH +/- buttons and strands the user. With the options
list shrinking from 11 entries to 5, any returning user whose saved `frontCount`
was 4/8/12/20/40/60/80/120/140/160 would be stranded — snapping guarantees a real
selected index.

The exact helper shape and tie-break-up implementation are Claude's discretion,
provided the result is always a member of `NK_FRONT_COUNT_OPTIONS`. Keep the
existing non-finite / non-positive fallback to `DEFAULT_NK_SETTINGS.frontCount`
(100) intact; the snap applies to the rounded valid value.

Update `src/storage/practices.test.ts` (and `src/storage/storage.test.ts` if it

frontCount (e.g. 40, 80, 120, 160, and a sub-100 value) snaps to the correct
