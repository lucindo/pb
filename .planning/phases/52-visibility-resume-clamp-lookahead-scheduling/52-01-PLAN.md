<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller → engine.topUpLookahead | Untrusted N (cue count); could be called with arbitrarily large args.cues array each rAF tick (~60 Hz) — would exhaust WebAudio scheduler. |
| caller → engine.cancelFutureCues | Repeated calls per tick during settings spam (mid-session BPM/timbre change) iterate activeCues unbounded times. |
| schedule() → activeCues Set | Cancel + topUp races: if a cue is mid-creation when cancel iterates the snapshot, the snapshot misses it; the new cue lives unbounded until pruneExpiredCues catches its cleanupAt. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-52-01 | Denial of Service | `topUpLookahead(args.cues)` | mitigate | The args.cues array is computed by Plan 03's `walkFutureCues` helper which is itself bounded by `LOOKAHEAD_MIN_CUES` + `LOOKAHEAD_WINDOW_SEC` + `targetSec`. Plan 03 caps cue count at ~max(LOOKAHEAD_MIN_CUES, ceil(LOOKAHEAD_WINDOW_SEC / minCycleSec)) ≈ ≤ ~12 cues even at 7 BPM. Plan 01 engine-side: NO additional cap (the caller is trusted within the codebase; adding a defensive cap here would hide a Plan 03 bug). Pre-existing `pruneExpiredCues()` keeps `activeCues` bounded over a long session. |
| T-52-02 | Denial of Service | `cancelFutureCues()` snapshot iteration | accept | Snapshot length bounded by `activeCues.size`, which is bounded by lookahead window (≤ ~12 cues). Worst case: 12 cancel() calls + 12 Set deletes per call. Cancel is itself bounded (per-osc stop + N disconnects, all O(1)). User-triggered (mute), not per-tick. Cost is negligible. |
| T-52-03 | Tampering | `cancel()` closure idempotency | mitigate | Per Task 2: every disconnect wrapped in try/catch (mirrors the existing 'ended' listener at cueSynth.ts L164-178). cancelScheduledValues + double stop() is safe. The 'ended' listener firing AFTER cancel() does another disconnect — also safe via try/catch. |
| T-52-SC | Tampering | npm/pip/cargo installs | n/a | DEPS-01: NO new runtime dependencies in Plan 01. No package install tasks. Legitimacy gate not required. |

**Note:** No new I/O boundaries, no new auth surface, no new PII handling, no new network calls. The audio scheduler is in-process and bounded by existing engine lifecycle (start → close). The change set is pure additive on existing closed seams.
