<objective>
Build `useNKEngine` — the Navi Kriya OM-counting session engine hook. It is a
self-rescheduling `setTimeout` chain that advances one OM per tick, drives the
front -> back -> next-round phase machine, and mirrors the display triple
`{phase, round, count}` into React state. Audio is injected as callbacks, so the
hook is audio-agnostic and fully testable with fake timers.

Purpose: this is the core technical engine of the phase. Every other Phase 31
plan consumes nothing from it directly (App.tsx wires it in Wave 2), but it must
exist and be correct first.

Output: `src/hooks/useNKEngine.ts` (new) + `src/hooks/useNKEngine.test.tsx` (new).
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| settings -> engine | `NaviKriyaSettings` passed into `start()` — already coerced at the storage boundary by `coerceNaviKriyaSettings` before reaching this hook |
| timer callback -> React | `setTimeout` callback mutates a ref and sets state; no external/untrusted input crosses here |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-31-01 | Tampering | `start()` receiving a non-multiple-of-4 `frontCount` (yields fractional backCount) | accept | `coerceNaviKriyaSettings` enforces multiple-of-4 at the storage boundary before settings reach the hook; the hook is downstream of that gate. No injection sink. |
| T-31-02 | Denial of Service | A pending `setTimeout` firing into a torn-down component on unmount | mitigate | `useEffect` cleanup calls `clearTimeout(timer.current)`; `end()` also clears the timer (RESEARCH Pitfall 4). |
| T-31-03 | Tampering | Stale-closure read of `cueOn`/phase inside `stepOm` producing wrong behavior | mitigate | All per-tick values read from `eng.current` (mutable ref); `toggleCue` mutates the ref directly. No closed-over state in `stepOm`. |
</threat_model>
