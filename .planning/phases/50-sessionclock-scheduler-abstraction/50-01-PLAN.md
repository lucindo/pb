If provided, `clock.schedule` forwards to it; if absent, `clock.schedule` is a typed no-op. The engine constructs `createAudioSessionClock(audioCtx, engineScheduleFn)` at the call-site in Plan 50-06. **Revision 2 — Blocker #1 (`notifySuspended()` engine-only escape hatch):** The plan-checker's iteration-2 verification confirmed that the pre-refactor `audioEngine.ts` L429-449 InvalidStateError catch block performs a SYNCHRONOUS state report by calling `opts.onStateChange?.(readState())` — NOT a state-change-event-driven fan-out. The AC was already `'suspended'` before `resume()` was called, stays `'suspended'` after the rejection, and no statechange event fires. The clock's listener cannot cover this case. Plan 50-04's revision-1 disposition (b) "the AC's natural statechange to 'suspended' after the rejection fires the clock's listener" was grounded in an incorrect assumption — this is the documented iOS Safari recovery path (Plan 06 D-38), NOT an edge case.

The fix: `createAudioSessionClock`'s return type is AUGMENTED to `SessionClock & { notifySuspended(): void }`. The `notifySuspended()` method synchronously invokes every callback in `suspendSubscribers` (same fan-out path the statechange listener uses on a real `'suspended'` event). This is an ENGINE-ONLY escape hatch — surfaced on the augmented return type for the engine's internal use. The public `SessionClock` interface read-only surface stays at exactly 6 members (`now`, `schedule`, `setMasterGain`, `onSuspend`, `onResume`, `onClose`). External consumers (useAudioCues, useSessionEngine, useNaviKriyaAudio, useAmbientScale, the engine's own `readonly clock: SessionClock` member on the AudioEngine interface) see only `SessionClock` — they cannot call `notifySuspended()`.
