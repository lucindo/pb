# Phase 05.1 Plan 02: Safari Outer-Ring Visual Gap Fix (D-10/D-12) Summary

**One-liner:** Negative-inset compensation (`inset: -1.5px`) on both `BreathingShapeBody` and `BreathingShapeLeadIn` outer-ring spans closes the Safari-only 1.5px gap at peak inhale without touching orb scale constants or theme.css.

## What Was Done

### Root Cause (Box-Model Analysis)

The outer reference ring span used `inset-0` (= `top:0; right:0; bottom:0; left:0`) with Tailwind preflight's global `border-box` sizing. The 1.5px `border-width` (from `theme.css .orb-ring--outer`) lives *inside* the 100% box. The orb at `scale(MAX_SCALE = 1.0)` fills the container to exactly 100%. On Safari (which follows the spec correctly), the ring's visible border edge sat 1.5px inside the container, creating a perceptible gap between the orb's outer edge and the ring at peak inhale. Chromium happened to render both edges coincident (non-spec behavior), masking the same arithmetic.

### Fix Applied

Both render sites were updated to replace `inset-0` (Tailwind class) with an explicit `style={{ inset: '-1.5px' }}`. This shifts the entire border-box outward by exactly the border width, so the border's outer edge sits 1.5px beyond the container's 100% box — matching the orb's `scale(1.0)` edge on Safari, Chromium, and Firefox.

### Files Changed

**1 file changed: `src/components/BreathingShape.tsx`**

**2 sites changed:**
1. `BreathingShapeBody` outer ring (~line 67-71 before, ~line 67-80 after): the primary render site for breathing sessions.
2. `BreathingShapeLeadIn` outer ring (~line 137-141 before, ~line 145-153 after): the lead-in countdown render site — required by D-12 so the gap does not reappear during the 3-2-1 countdown.

Both spans are now byte-identical:
```tsx
<span
  aria-hidden="true"
  className="orb-ring--outer absolute rounded-full border-solid"
  style={{ inset: '-1.5px' }}
/>
```

## Discretion Pick #2: Negative Inset over Content-Box

The plan specified two acceptable mechanisms (D-10). Negative inset (option 2) was chosen over `box-sizing: content-box + width/height 100%` (option 1) for these reasons:

- **Localized to JSX** — no theme.css edit (smaller blast radius; zero CSS file changes).
- **Single source of truth**: `style={{ inset: '-1.5px' }}` consumes the `1.5px` border-width constant verbatim from the same logical location. Option (1) would require a paired `box-sizing: content-box` in theme.css plus width/height declarations, splitting the contract across two files.
- **Respects Tailwind preflight convention**: Tailwind sets `box-sizing: border-box` globally; option (1) would require a per-rule opt-out that drifts from project convention.
- **Cross-browser verified**: The inset/translate idiom (used for the inner ring's `left-1/2 top-1/2 -translate-x-1/2 -translate-y-1/2` pattern) is known to render identically on Safari, Chromium, and Firefox from Phase 2 Plan 02-02.
