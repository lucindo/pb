## Orb Scale Fix Approach

### Question 1 — Fix mechanism

| Option | Description | Selected |
|--------|-------------|----------|
| Bump MAX_SCALE buffer (e.g. 1.02) | Smallest delta. Change MIN_SCALE/MAX_SCALE/MID_SCALE constants + matching CSS tokens. Risk: visual shift on Chromium where the gap doesn't exist (orb would now overflow ring slightly). Also affects MID_SCALE via the average formula. | |
| Compensate outer ring sizing | Outer ring uses Tailwind `inset-0` + `border-solid` (default `box-sizing: border-box`). ## Round 2 — Plan 06 Discussion (2026-05-10)

**Trigger:** Plan 01 audio-resume fix shipped per spec but empirically FAILED on real iPhone iOS Safari. Diagnostic spike captured (Web Inspector instrumentation, reverted post-capture) revealed two root causes the original spec did not anticipate:

1. iOS Safari uses a WebKit-only `'interrupted'` AudioContext state (not in MDN spec enum).
2. `audioCtx.resume()` from a `visibilitychange` handler is not a user-gesture context on iOS Safari; rejects with `InvalidStateError "Failed to start the audio device"`.

Plan 01 D-09 silent-absorbed the rejection, so audio dies invisibly while visuals continue.

Plan 06 was discussed under the diagnostic-grounded constraints from `.continue-here.md`'s two blocking anti-patterns (jsdom ≠ Safari; spec-not-device).

### Areas Discussed

#### Area: Recovery affordance shape
- **Question:** "Post-unlock, audio dead, visuals continue. What does user see/do to recover?"
- **Options presented:**
  1. Explicit "Tap to resume audio" control (Recommended)
  2. Passive: any session-area tap retries resume()
  3. Silent, document as known issue
  4. Banner/toast across top
- **User chose:** Explicit affordance.
- **Captured as:** D-29.

- **Question:** "After successful resume tap, what happens to the affordance?"
- **Options presented:**
  1. Auto-hide once resume succeeds (Recommended)
  2. Stays until next phase boundary
  3. Stays for fixed duration (1s) then fades
- **User chose:** Auto-hide on `state → 'running'`.
- **Captured as:** D-30.

#### Area: Mute-toggle as recovery shortcut
- **Question:** "If user un-mutes via mute toggle during needs-resume state, what happens?"
- **Options presented:**
  1. Un-mute triggers resume + clears flag (Recommended)
  2. Un-mute stays pure (mute flag only)
  3. Un-mute auto-triggers resume only if currently muted-AND-needs-resume
- **User chose:** Un-mute triggers resume + clears flag.
- **Captured as:** D-31.

#### Area: Affordance placement (UI slot)
- **Question:** "Where does the 'Tap to resume audio' affordance live in the UI?"
- **Options presented:**
  1. Replace mute icon glyph temporarily (Recommended)
  2. Floating button below orb
  3. Inline below breathing orb (non-floating)
  4. Persistent toolbar slot
- **User chose:** MuteToggle slot morphs (icon + aria-label) during `needs-resume`.
- **Captured as:** D-32, D-32a (reduced-motion: instant state swap), D-32b (a11y: aria-live consideration left to planner).

#### Area: AC-closed terminal fallback
- **Question:** "User taps resume affordance, resume() rejects again (AC closed not interrupted). What now?"
- **Options presented:**
  1. Reconstruct engine on the same gesture (Recommended)
  2. Prompt user to End + Restart session
  3. Keep retrying resume(); don't reconstruct
  4. Reconstruct silently on first failed resume, no separate fallback path
- **User chose:** Reconstruct engine on same gesture.
- **Captured as:** D-33, D-34 (new public hook surface `resume()` + `audioStatus`), D-35 (anchor RE-anchor on reconstruction only, not on plain resume), D-35a (no lead-in replay), D-35b (mute state survives reconstruction).

### Decisions Made by Claude (within bounded discretion)

- D-36: State tracking mechanism (`statechange` listener vs post-call polling) — planner picks.
- D-37: `'interrupted'` treated symmetrically with `'suspended'` in state-machine reads.
- D-38: D-09 silent-absorb relaxed ONLY for `InvalidStateError` from `resume()` — sets `audioStatus = 'needs-resume'`. All other errors stay silent.
- D-39: Visibility-handler still optimistically calls `resume()` — sometimes succeeds without gesture; affordance only appears on rejection.
- D-40: vitest.setup.ts FakeAudioContext extended with `'interrupted'` state, `_simulateInterrupted`, `_simulateResumeReject`, `dispatchStateChange`.
- D-41: Hook tests cover four-state matrix.
- D-42: Integration tests cover MuteToggle morph + dual-path tap behavior.
- D-43: Real-device UAT scenarios.
- D-44: Cross-browser regression posture.

### Anti-Pattern Mitigations Applied

- **"Treating jsdom green as Safari green"** → D-43 mandates real-device UAT before sign-off; Plan 06 cannot close on auto-suite green alone.
- **"Spec'ing iOS Safari behavior from MDN/spec without device confirmation"** → Plan 06 spec is grounded in the captured diagnostic in `05.1-UAT.md` Task 2. Any further iOS behavior the planner needs (e.g., addEventListener('statechange') behavior during lock; createAudioEngine() success from synchronous tap-handler chain) requires an inline diagnostic spike, not spec-only inference.

### Deferred (Plan 06 round, in addition to prior rounds)

- Animated fade on affordance hide (current spec: instant; can be revisited if UAT feels abrupt).
- Wake Lock + audio coupling — Plan 06 leaves them independent. The wake lock listener is separately gated and untouched.
- Telemetry / analytics on resume-needed events — v1 has no telemetry surface; revisit if v2 adds one.
- Closed-AC reconstruction when audio was muted at lock time (D-35b says mute state survives; no special UX). May want a subtle "audio ready" cue if pattern shows up in UAT, but not for v1.
- Cross-app interruption (phone call mid-session) — same code path as lock-screen (`'interrupted'` state), so the Plan 06 design handles it; UAT scenario could be added but is not the primary case.
