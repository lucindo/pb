**Result at near-peak inhale:** **FAIL** — visible gap ≈ 75 px on each side between orb edge and outer ring inner edge.

**iOS Safari (iPhone) result:** **PASS** — orb meets ring; no gap. The bug is Safari Desktop only.

**Inspector data captured at progress=0.971:**

| Element | getBoundingClientRect().width | Expected |
|--------|-------------------------------|----------|
| `div[role="img"][data-phase]` (container) | 360.00 px | 360 px ✓ |
| `.orb-ring--outer` | 363.00 px | 363 px ✓ (`inset:-1.5px` resolves correctly) |
| `.orb` | **209.00 px** | 360 px ✗ — sized at 58% of parent (≈ MIN_SCALE × 360) |

The orb's `transform: scale(0.988016)` (matches `0.58 + 0.971 × 0.42`) is mathematically correct, but the underlying box is collapsed to 209 px on Safari Desktop instead of filling the 360 px container via `inset-0`.

**Root cause hypothesis (matches existing WR-03 comment in `BreathingShape.tsx:82`):**
Absolutely-positioned children of a `display: grid; place-items: center` parent are inconsistently sized on Safari Desktop. The inner ring already uses an explicit `left:50%; top:50%; width: 58%; height: 58%; -translate-{x,y}-1/2` pattern to dodge this. The `.orb` div and the outer-ring span still rely on `inset-0` / `inset: -1.5px` shorthand, which Safari Desktop honors for the outer ring (negative inset forces non-zero size) but NOT for `.orb` (where `inset: 0` collapses the box to 58% of parent, matching the inner ring's explicit size — most likely Safari is using sibling auto-track sizing).

**Plan 05.1-02 status:** addressed the 1.5px border-alignment surface symptom (real but secondary). The deeper WR-03-class root cause remains. Re-fix required.
