### Diagnostic findings (Web Inspector console capture, 2026-05-10)

Temporary `console.log` instrumentation added to `useAudioCues.onVisibility` and `audioEngine.resume()` (later reverted; not committed). Mac+iPhone Safari Web Inspector session captured the following sequence on a single lock(~10s)/unlock cycle:

```
visibilitychange fired → visibilityState: "hidden",  hasEngine: true
                          ↓ (lock duration)
visibilitychange fired → visibilityState: "visible", hasEngine: true
engine.resume() entry  → closedFlag: false,  audioCtx.state: "interrupted",  currentTime: 21.152
engine.resume()        → REJECTED  InvalidStateError "Failed to start the audio device"
                                   audioCtx.state after reject: "suspended"
```

**Key facts:**

1. `visibilitychange` listener wires up and fires correctly on both hide and visible — D-01..D-09 gate logic is sound.
2. `audioCtx.state === "interrupted"` is a **WebKit-only non-standard state** not present in MDN's published AudioContext state enum (`suspended | running | closed`). Plan 01 spec assumed lock → `suspended`; reality is lock → `interrupted` → (after rejected resume()) `suspended`.
3. `audioCtx.resume()` invoked from a `visibilitychange` handler is **NOT considered a user gesture** by iOS Safari. The promise rejects with `InvalidStateError: "Failed to start the audio device"` even though the AC transitions to `suspended` (which is normally recoverable from a gesture). The D-09 silent-absorb pattern hides this rejection, so visuals continue while audio is dead — no surface to the user.
4. The mute-toggle "recovery path" the user instinctively tried does not work because `setMuted(false)` only flips a flag (`audioEngine.ts:38`); it never calls `resume()`. A user tap on the mute button IS a real user gesture, so this path could trivially be made to succeed if `setMuted(false)` invoked `resume()` synchronously.

**Implication for Plan 05.1-06 spec (input only; binding spec to be written via /gsd-plan-phase):**

| # | Direction |
|---|-----------|
| 1 | Treat `interrupted` symmetrically with `suspended` in any state-machine surface. Both are recoverable from a user gesture; neither is a terminal state. |
| 2 | The visibilitychange path cannot be the sole recovery — it lacks user-gesture context. UI must surface a "Tap to resume audio" affordance when post-unlock `resume()` rejects. |
| 3 | `setMuted(false)` is the cheap gesture-attached shortcut: have it call `resume()` synchronously so users who instinctively un-mute recover for free. |
| 4 | Use `audioCtx.addEventListener('statechange')` to drive the recovery flag transitions cleanly instead of polling. |
| 5 | Relax D-09 specifically for `InvalidStateError` on iOS — propagate to a status surface (e.g. `audioStatus: 'needs-resume'`) rather than swallowing. Keep silent-absorb for unknown errors. |
| 6 | Tests need a mock AC `'interrupted'` state value (jsdom default does not expose). ~6 new tests; existing D-01..D-09 cases stay green if the gate is widened additively. |

**Carry-forward (Plan 01 was not wasted):** D-01..D-09 gate scaffolding stands; Plan 06 widens it. The empirical failure here drove the diagnosis; the alternative (shipping a guessed engine-reconstruction Plan 06 preemptively) would have likely missed the user-gesture half of the bug entirely.

**Root cause hypothesis (matches existing WR-03 comment in `BreathingShape.tsx:82`):**
Absolutely-positioned children of a `display: grid; place-items: center` parent are inconsistently sized on Safari Desktop.
