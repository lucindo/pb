# Phase 05.1 Plan 06: iOS Safari AudioContext Recovery - Research

**Researched:** 2026-05-10
**Domain:** Web Audio API on iOS Safari — non-standard `'interrupted'` state recovery, user-gesture-attached resume, engine reconstruction fallback
**Confidence:** HIGH (device-confirmed diagnostic data + WebKit spec + cross-library prior art)

## Summary

Phase 5.1 Plan 01 implemented `audioCtx.resume()` on `visibilitychange→visible` per Phase 5 D-03's wake-lock pattern. Real-device UAT (`05.1-UAT.md` Task 2) proved the spec was insufficient on iOS Safari for two reasons that the Plan 01 spec couldn't have foreseen from MDN alone:

1. **iOS Safari uses a non-standard `'interrupted'` state**, not the spec's `'suspended'`. The state IS distinguished and exposed; the AC transitions back to `'suspended'` only AFTER a failed resume attempt.
2. **`audioCtx.resume()` invoked from a `visibilitychange` handler is not a user gesture on iOS**, so the resume rejects with `InvalidStateError: "Failed to start the audio device"`. Plan 01's D-09 silent-absorb hides this rejection — audio dies invisibly.

The CONTEXT.md re-plan section (D-29..D-44, committed 421f9ab) already specifies a four-layer recovery: (a) optimistic visibility-handler resume; (b) `audioStatus: 'needs-resume'` flag surfaced to App.tsx; (c) MuteToggle morphs into a user-tappable resume affordance — the gesture-attached recovery path the visibility handler cannot supply; (d) if gesture-attached `resume()` still fails (closed AC, hardware veto), reconstruct the engine inside the same gesture chain.

Plan 01-05 locked decisions D-01..D-28 carry forward unchanged. Plan 06 extends — does not rewrite — the existing scaffolding.

**Primary recommendation:** Implement the CONTEXT.md D-29..D-44 spec verbatim. The interrupted-state recovery pattern is well-established in Tone.js / Howler.js / standardized-audio-context, and the reconstruction-on-failure fallback is the empirically-proven last resort (already validated in this codebase by the user's observation that "ending and restarting the session works").

### Pattern 1: Statechange listener replaces polling (D-36)

**What:** Engine wires a single `audioCtx.addEventListener('statechange', ...)` at construction; pushes state into a callback supplied by the hook.

**When to use:** Whenever the React state surface needs to react to AC state transitions driven outside the hook (e.g., iOS auto-suspend).

**Example:**
```typescript
// Source: MDN BaseAudioContext.state + project Phase 5 D-04 callback pattern
// In audioEngine.ts createAudioEngine factory:
export async function createAudioEngine(opts: { onStateChange?: (state: AudioContextState) => void } = {}): Promise<AudioEngine> {
  const audioCtx = new AudioContext()
  if (audioCtx.state === 'suspended') { /* WR-06 path unchanged */ }

  // Plan 06 D-36: wire statechange. Single listener per engine instance; cleared by close().
  // The cast to `AudioContextState | 'interrupted'` reflects WebKit's superset (D-37).
  const onStateChange = (): void => {
    opts.onStateChange?.(audioCtx.state)
  }
  audioCtx.addEventListener('statechange', onStateChange)

  // ... existing engine assembly ...

  // Existing close() teardown removes the listener too:
  async close(): Promise<void> {
    if (closed) return
    closed = true
    audioCtx.removeEventListener('statechange', onStateChange)
    await audioCtx.close()
  }
}
```

### Pattern 2: Gesture-attached recovery chain (D-33)

**What:** When a user gesture (tap) is the only path back to `'running'` from `'interrupted'`, dispatch `resume()` synchronously inside the click handler. If resume rejects, reconstruct in the same handler — the iOS gesture context spans the synchronous-call chain.

**When to use:** Any time `resume()` from a non-gesture context (visibilitychange, statechange, setInterval) has failed.

**Example:**
```typescript
// Source: github.com/Tonejs/Tone.js/pull/716 + WebKit bug 263627 workaround
// In App.tsx onMuteToggle handler (extends existing persistedSetMuted line 142-145):
const onMuteClick = useCallback(async () => {
  if (audio.audioStatus === 'needs-resume') {
    // Gesture is HERE — chain everything synchronously.
    try {
      await audio.resume()
    } catch {
      // Plain resume failed → reconstruct in same gesture chain.
      // useAudioCues.resume() internally handles the close+new-create + re-anchor.
    }
  }
  persistedSetMuted(!audio.muted)
}, [audio, persistedSetMuted])
```

### Pattern 3: Reconstruction with state preservation (D-33 / D-35 / D-35b)

**What:** When `resume()` cannot recover the AC (state stuck, AC closed, hardware veto), close the old engine and construct a new one. Replay mute state. Re-anchor the dual-anchor against the new AC's `currentTime` (which starts at 0).

**Example sequence:**
```typescript
// Source: Phase 3 D-09/D-11 (existing close lifecycle) + Phase 5.1 D-33/D-35
// Inside useAudioCues.resume() public method when plain resume fails:
async function reconstructEngine(): Promise<void> {
  const oldEngine = engineRef.current
  const currentMuted = muted
  engineRef.current = null // synchronously null per Pitfall 3 pattern

  // Step 1: close old engine (D-11 idempotent close)
  if (oldEngine !== null) {
    await oldEngine.close().catch(() => undefined) // D-09 silent
  }

  // Step 2: new engine — MUST be in same gesture chain on iOS
  try {
    const newEngine = await createAudioEngine({ onStateChange: handleStateChange })
    engineRef.current = newEngine
    newEngine.setMuted(currentMuted) // D-35b
    // Step 3: signal re-anchor to App.tsx via callback prop or new event
    onReanchorRequiredRef.current?.(newEngine.now())
    setAudioStatus('ok')
  } catch {
    // D-10 terminal fallback
    setAudioStatus('unavailable')
    setAudioAvailable(false)
  }
}
```

### Pattern 4: Symmetric interrupted/suspended treatment (D-37)

**What:** Both `'interrupted'` and `'suspended'` are non-terminal recoverable states. The `audioStatus = 'needs-resume'` flag fires for either, AND only AFTER a prior resume attempt has rejected (avoids flashing the affordance during the brief transition window).

**Example state machine:**
```
AC state          + prior resume attempt   → audioStatus
─────────────────────────────────────────────────────────
'running'         (n/a)                    → 'ok'
'suspended'       not attempted            → 'ok' (visibility handler will try first)
'suspended'       rejected                 → 'needs-resume'
'interrupted'     not attempted            → 'ok' (visibility handler will try first)
'interrupted'     rejected                 → 'needs-resume'
'closed'          (n/a)                    → 'unavailable'
```

### Anti-Patterns to Avoid

- **Polling `audioCtx.state` on rAF/setInterval (D-36 rejects):** Wastes battery on a locked phone, defeats the wake-lock budget. Use `addEventListener('statechange')`.
- **Calling `resume()` from non-gesture contexts and assuming it succeeds:** Confirmed by device diagnostic to reject with InvalidStateError. The optimistic call in the visibility handler (D-39) is preserved because it sometimes works (headphone in, brief lock, certain iOS versions), but the codepath must NOT depend on success.
- **Silent-absorbing `InvalidStateError` (Plan 01 D-09's blanket posture):** Causes the exact bug Plan 06 fixes. D-38 narrows the absorb posture: this specific error becomes a UI signal.
- **Re-anchoring on plain resume (Phase 3 dual-anchor violation):** D-06 forbids it because plain resume preserves the AC clock. Only reconstruction needs re-anchor (new AC = new clock origin at 0).
- **Replaying the lead-in on reconstruction (D-35a forbids):** Mid-session, the user does not need 3-2-1 again. Schedule the next In/Out cue at the next phase boundary, same as Plan 01 D-05.
- **Adding `console.error` on resume rejection:** Project convention discretion #4 — no console.debug/error in fallback paths. The state surface IS the diagnostic.
- **Spec'ing iOS Safari behavior from MDN without device confirmation (Anti-pattern from .continue-here.md):** Plan 06's spec is grounded in device-confirmed data. Any further behavior assumption (e.g., whether reconstruction inside a tap handler reliably succeeds) MUST be verified by inline diagnostic spike before shipping.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| State tracking for AC | rAF/interval polling of `audioCtx.state` | `addEventListener('statechange')` (D-36) | Battery drain; misses transitions between polls; conflicts with calm/hands-off ethos |
| Cross-browser state enum | Branch on user-agent string for `'interrupted'` | Treat `'interrupted'` symmetrically with `'suspended'` (D-37) | UA sniffing is brittle; symmetric treatment works on every browser (non-WebKit never enters `'interrupted'`) |
| Resume retry loop | `setTimeout(() => resume(), n)` cascade | Gesture-attached recovery + reconstruction fallback (D-33) | No amount of non-gesture retries beats iOS's autoplay policy; only a gesture handler can succeed |
| Visual regression detection for the affordance morph | jsdom snapshot of glyph SVG paths | Structural assertion on aria-label change + glyph presence (D-42) + real-device UAT (D-43) | jsdom doesn't render visually; structural is the most jsdom can verify; visual fidelity is a UAT concern |
| Re-anchor logic in audioEngine | Engine tracks dual-anchor internally | Hook signals App.tsx; App.tsx owns audioAnchorRef (D-35 + D-02 boundary) | Phase 3 D-09→D-11 boundary: engine is service, App.tsx owns the dual-anchor invariant |
| Keep-alive silent oscillator to avoid `'interrupted'` | Continuous tiny-gain oscillator | Reconstruction-on-failure (D-33) | Tone.js considered and rejected; battery drain on locked phone defeats Phase 5 wake lock |

**Key insight:** The Web Audio iOS quirk has been litigated for ~5 years across multiple major libraries (Tone.js issues #341, #767, PR #716; Howler.js issues #1031, #1287, #1559, #1702). The settled prior-art recovery patterns are: (1) gesture-attached resume, (2) reconstruction on persistent failure, (3) `'interrupted'` treated symmetrically with `'suspended'`. Plan 06's spec aligns with all three.

## Common Pitfalls

### Pitfall 1: Calling `resume()` from a non-gesture context and treating success as the common case

**What goes wrong:** Code assumes the optimistic `audioCtx.resume()` call from the visibility handler will succeed; downstream UI logic doesn't account for the rejection path; user experiences silent audio death.

**Why it happens:** MDN documents `resume()` returning `Promise<void>` without explicit iOS-quirk callouts. Plan 01 D-09 blanket silent-absorb was the wrong polarity for THIS specific failure mode.

**How to avoid:** Treat the optimistic call as a cheap try; ALWAYS wire the gesture-attached fallback path. Watch for `InvalidStateError` specifically (D-38).

**Warning signs:** Test suite passes; real-device audio silently dies after lock/unlock; no error in console.

### Pitfall 2: Reconstruction inside the wrong execution context

**What goes wrong:** Closing the old AC and creating a new one inside `setTimeout` / `requestAnimationFrame` / a `.then()` callback breaks the iOS gesture chain. New AC starts in `'interrupted'` immediately.

**Why it happens:** iOS Safari's "user gesture" is a synchronous call-stack property — it does NOT propagate through `setTimeout`, `Promise.then`, or `requestAnimationFrame` callbacks. Verified empirically across the Web Audio ecosystem.

**How to avoid:** All reconstruction logic must be synchronous (or `await`-chained) from the click handler. The `await audio.resume()` followed by `await audio.reconstruct()` chain inside the click handler is acceptable — both promises resolve/reject within the same gesture context.

**Warning signs:** Reconstruction "works" in vitest but `'interrupted'` is the immediate state on the new AC in real Safari.

### Pitfall 3: Forgetting to replay mute state on reconstruction (D-35b)

**What goes wrong:** User had muted audio mid-session, locks phone, unlocks, taps affordance, audio comes back AT FULL VOLUME instead of staying muted.

**Why it happens:** New AC starts in default state (muted=false per Phase 3 D-07); the engine's internal `muted` flag resets along with everything else.

**How to avoid:** Capture `muted` before close(); after `createAudioEngine()` resolves, call `newEngine.setMuted(currentMuted)` synchronously BEFORE returning from reconstruct(). The React `muted` state is also preserved because `useAudioCues` state is in React, not in the engine.

**Warning signs:** Integration test in D-42 must verify `engine.setMuted(true)` is called on the new engine when reconstruction happens from a muted state.

### Pitfall 4: Re-anchoring on plain resume (D-06 violation)

**What goes wrong:** Plain `audioCtx.resume()` resumes the same AC — `currentTime` continues from where it was. If the dual-anchor is reset to `audioCtx.currentTime` at this moment, every subsequent cue schedules against the wrong origin and fires at the wrong time.

**Why it happens:** Confusing "AC was suspended" with "AC was recreated." They have different anchor semantics.

**How to avoid:** ONLY re-anchor on reconstruction (D-35). The plain-resume path leaves audioAnchorRef untouched (D-06).

**Warning signs:** Cues fire several seconds early/late after recovery; integration test should assert `audioAnchorRef.current` is unchanged after plain resume and CHANGED after reconstruction.

### Pitfall 5: Affordance flashes briefly during normal transitions

**What goes wrong:** User taps Start, lead-in begins, brief AC state churn fires the affordance for a frame before everything settles.

**Why it happens:** AC startup can transition through `'suspended'` momentarily (Phase 3 WR-06 path); naive `audioStatus = state === 'suspended' ? 'needs-resume' : 'ok'` triggers immediately.

**How to avoid:** Gate `audioStatus = 'needs-resume'` on BOTH (state is suspended/interrupted AND a prior visibility-resume attempt rejected). Not just state alone. Pattern 4 in this doc enforces this.

**Warning signs:** Affordance appears for one render frame during startup; user reports "flicker" on a real device.

### Pitfall 6: aria-live spam on every state transition

**What goes wrong:** Screen reader announces "Resume audio" / "Mute audio cues" rapidly as state churns.

**Why it happens:** `aria-live="polite"` announces every change to the labeled element when the live-region wraps the button.

**How to avoid:** Live region announces the TRANSITION (e.g., "Audio paused, tap to resume") only when it enters `needs-resume`, not on every render. Use a separate visually-hidden `<div role="status">` in App.tsx that updates only on state-change (planner discretion #6).

**Warning signs:** Manual screen-reader UAT reports rapid-fire announcements.

### Pitfall 7: jsdom can't reproduce the InvalidStateError lifecycle

**What goes wrong:** Test suite green; real-device still broken.

**Why it happens:** jsdom + FakeAudioContext can ONLY model what we tell it to. The `_simulateResumeReject(errorName)` test hook (D-40) lets us inject the rejection; without it, tests can't exercise the failure path.

**How to avoid:** D-40 mandates `_simulateInterrupted()` + `_simulateResumeReject(errorName)` + `dispatchStateChange()` on FakeAudioContext. D-41 mandates the four-state matrix. D-43 mandates real-device UAT as a blocking gate — jsdom green is necessary but not sufficient.

**Warning signs:** Anti-pattern from `.continue-here.md`: "Treating jsdom green as Safari green." Plan 06 explicitly mitigates this.

```typescript
// Source: project pattern + WebKit statechange spec
// File: src/audio/audioEngine.ts (extends existing createAudioEngine)
export interface AudioEngineOptions {
  /** Plan 06 D-36: receives every audioCtx.state transition. The hook listens and
   *  pushes the value into React state. Engine stays React-free per D-02. */
  onStateChange?: (state: AudioContextState | 'interrupted') => void
}

export async function createAudioEngine(opts: AudioEngineOptions = {}): Promise<AudioEngine> {
  const audioCtx = new AudioContext()
  if (audioCtx.state === 'suspended') {
    try { await audioCtx.resume() } catch (err) {
      await audioCtx.close().catch(() => undefined)
      throw err
    }
  }

  const onStateChange = (): void => {
    // Cast tolerates WebKit's 'interrupted' superset (D-37).
    opts.onStateChange?.(audioCtx.state as AudioContextState | 'interrupted')
  }
  audioCtx.addEventListener('statechange', onStateChange)

  // ... rest of engine assembly unchanged ...

  const engine: AudioEngine = {
    // ... existing methods ...
    async close(): Promise<void> {
      if (closed) return
      closed = true
      audioCtx.removeEventListener('statechange', onStateChange)
      await audioCtx.close()
    },
    async resume(): Promise<void> {
      if (closed) return
      try {
        await audioCtx.resume()
      } catch (err) {
        // D-38 (relaxed D-09): InvalidStateError surfaces; others silent-absorb.
        if ((err as DOMException)?.name === 'InvalidStateError') {
          // Signal explicitly — onStateChange may also fire as AC transitions
          // 'interrupted' → 'suspended' post-reject (per device diagnostic).
          // Either path leads to audioStatus='needs-resume' in the hook.
          opts.onStateChange?.(audioCtx.state as AudioContextState | 'interrupted')
        }
        // Else: silent-absorb (D-09 default posture preserved for unknown errors).
      }
    },
  }
  return engine
}
```

```typescript
  // current suspend cycle. Prevents Pitfall 5 (affordance flashes during startup).
  const visibilityResumeAttemptedRef = useRef<boolean>(false)

  // audioStatus transitions.

      if (engineRef.current.now() === 0 /* surrogate — see audioCtx.state read */) {
        await reconstructEngine()
      }
    } catch {
      await reconstructEngine()
    }
  }, [/* reconstructEngine */])

  const reconstructEngine = useCallback(async (): Promise<void> => {
    // ... see Pattern 3 ...
  }, [muted])

  return {
    // ... existing fields ...
    audioStatus,
    resume,
  }
}
```

### Example 5: FakeAudioContext extension for tests

```typescript
// Source: existing vitest.setup.ts lines 164-203 + Plan 06 D-40
// File: vitest.setup.ts (extension)
class FakeAudioContext {
  state: AudioContextState | 'interrupted' = 'running'  // D-40: superset
  sampleRate = 44100
  destination = new FakeAudioNode()
  private _start = performance.now() / 1000
  private _listeners = new Map<string, Set<EventListener>>()
  private _resumeRejection: { name: string; message: string } | null = null

  get currentTime() { return performance.now() / 1000 - this._start }
  constructor(_options?: AudioContextOptions) {}

  createOscillator() { return new FakeOscillatorNode() }
  createGain() { return new FakeGainNode() }
  createBiquadFilter() { return new FakeBiquadFilterNode() }

  resume = vi.fn(async () => {
    if (this._resumeRejection !== null) {
      const err = new DOMException(this._resumeRejection.message, this._resumeRejection.name)
      this._resumeRejection = null
      // Per device diagnostic: 'interrupted' transitions to 'suspended' after rejected resume.
      if (this.state === 'interrupted') this.state = 'suspended'
      this._fireStateChange()
      throw err
    }
    this.state = 'running'
    this._fireStateChange()
  })

  suspend = vi.fn(async () => { this.state = 'suspended'; this._fireStateChange() })
  close = vi.fn(async () => { this.state = 'closed'; this._fireStateChange() })

  // Plan 01 D-14:
  _simulateSuspend = (): void => { this.state = 'suspended'; this._fireStateChange() }

  // Plan 06 D-40 additions:
  _simulateInterrupted = (): void => {
    this.state = 'interrupted'
    this._fireStateChange()
  }
  _simulateResumeReject = (errorName: string = 'InvalidStateError'): void => {
    this._resumeRejection = { name: errorName, message: 'Failed to start the audio device' }
  }
  dispatchStateChange = (): void => { this._fireStateChange() }

  addEventListener = vi.fn((type: string, listener: EventListener): void => {
    let set = this._listeners.get(type)
    if (!set) { set = new Set(); this._listeners.set(type, set) }
    set.add(listener)
  })
  removeEventListener = vi.fn((type: string, listener: EventListener): void => {
    this._listeners.get(type)?.delete(listener)
  })

  private _fireStateChange(): void {
    const evt = new Event('statechange')
    for (const l of this._listeners.get('statechange') ?? []) l(evt)
  }
}
```

## Open Questions

1. **Glyph choice for the resume affordance (Discretion #1):**
   - What we know: must be visually distinct from speaker-on/muted glyphs; must feel calm/meditative; no red/alarm semantics; reuses the 44×44 hit area.
   - What's unclear: which specific icon best telegraphs "tap to resume audio" without looking like a play button (which implies "start media").
   - Recommendation: refresh-arrow (suggesting "reload audio") — distinct from speaker glyphs, distinct from play-triangle, calm geometric shape. Backup option: speaker glyph with a small spark/dot indicator. Defer to the planner's discretion based on visual design fit; the structural test (D-42) cares about glyph presence + aria-label, not which specific path-d string.

2. **Engine reconstruction site (Discretion #2):**
   - What we know: D-33/D-34 mandate that reconstruction lives inside `useAudioCues.resume()` and uses `createAudioEngine()` under the hood. Both factory-method-on-engine and hook-level-orchestration satisfy.
   - What's unclear: ergonomics — does adding `reconstruct()` to the engine API add value, or does the hook orchestrating close+create stay cleaner?
   - Recommendation: hook-level orchestration. The engine has no need for an internal "rebuild" method — close+new-create is the natural lifecycle. The hook is the resource owner; reconstruction is a resource-management concern. Keeps the engine API minimal (one new option: `onStateChange` callback).

3. **State surface shape (Discretion #3):**
   - What we know: `audioStatus: 'ok' | 'needs-resume' | 'unavailable'` enum OR `needsResume: boolean` + existing `audioAvailable: boolean`.
   - What's unclear: which interface feels cleaner to App.tsx and to test code.
   - Recommendation: enum. Three states, one field — eliminates the impossible `(needsResume=true, audioAvailable=false)` combination. App.tsx pattern: `audio.audioStatus === 'needs-resume'`. Same compact shape as session.status.

4. **Re-anchor seam (Discretion #5):**
   - What we know: D-35 mandates re-anchoring of the dual-anchor on reconstruction. Must originate from `useAudioCues` (which knows the engine was reconstructed) and reach `App.tsx` (which owns `audioAnchorRef`).
   - What's unclear: optional callback prop on the hook OR refactor existing lead-in-completion path.
   - Recommendation: optional callback prop. App.tsx passes `onReanchorRequired={(newAudioTime) => { audioAnchorRef.current = newAudioTime; ... }}` to `useAudioCues`. Minimal seam, no churn to existing lead-in code. Hook fires the callback exactly once per reconstruction; null in test contexts that don't care.

5. **aria-live placement (Discretion #6):**
   - What we know: D-32b leaves this open; MAY be deferred to UAT.
   - What's unclear: at MuteToggle level OR App.tsx-level visually-hidden region.
   - Recommendation: App.tsx-level. Single `<div role="status" aria-live="polite" className="sr-only">` outputs "Audio paused, tap to resume" only on the `'ok' → 'needs-resume'` transition (debounced via useEffect dependency on audioStatus). Avoids Pitfall 6 (announcement spam) and keeps MuteToggle stateless.
