# Phase 8 — UI Design Contract

> Visual and interaction contract for Phase 8 (Storage Forward-Compat & Cross-Tab UI Sync).
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase UI Scope — read first

Phase 8 is a **storage-adapter behavior change** (STORAGE-01 read preserves on-disk `version`; STORAGE-02 write refuses downgrade) plus **one UI-observable behavior** (STORAGE-03: an App-level `storage` event listener calls `setStats(loadStats())` so the existing stats footer in another tab updates when this tab writes the envelope).

**There are NO new visual components, NO new spacing/typography/color tokens, NO new copy strings, NO motion changes, NO new accessibility surfaces, NO new responsive concerns.** The entire visual surface (`StatsFooter.tsx` in particular) is unchanged byte-for-byte. The only observable diff is that the existing `<p>{formatSessionCount(stats.totalSessions)} · {formatTotalMinutes(...)} total</p>` and `<span>{lastLine}</span>` text nodes may re-render with new numbers when the other tab writes — same DOM, new text content, no layout shift, no class changes, no animation.

Everything below confirms "no change" against the six design dimensions and locks the cross-tab-refresh interaction contract.

## Motion

**Phase 8 introduces no motion.** The cross-tab refresh:

- No CSS transition on the footer text nodes (existing footer has none).
- No fade/scale animation when numbers change.
- No reduced-motion branch needed (no animation to gate).
- React's default DOM diff updates the text content synchronously; users see the new value on the next paint with no transient state.

This is intentional: the cross-tab refresh is **decorative consistency**, not a notification. A flash or animation would falsely imply "something new happened in this tab" when in fact the change happened in another tab. Locked.

## Accessibility

**Phase 8 introduces no new accessibility surfaces.** Explicit dimension-by-dimension confirmations:

### Cross-tab footer update — no `aria-live` announcement
**Decision:** The footer text values that update via the storage-event listener are NOT wrapped in an `aria-live` region. Rationale:

- The stats footer is **persistent decorative content**, not a notification. It is always on-screen on the idle view (when `totalSessions > 0`); cross-tab updates are passive consistency, not events the user needs alerted to.
- Announcing "3 sessions, 47 min total" via screen reader every time another tab finishes a session would be **disruptive noise** — the user did not perform an action in this tab and is not waiting for feedback.
- The behavior matches in-tab updates: when the user clicks Reset in *this* tab, the footer updates without an aria-live announcement (existing Phase 4 contract). The cross-tab update should match in-tab semantics — same DOM target, same lack of announcement.
- WCAG 2.2 SC 4.1.3 (Status Messages) is not violated: this is not a status message in response to a user action in the current tab.

This decision is **documented** rather than skipped per the agent prompt's accessibility guidance: the dimension was considered, the answer is "no aria-live", and the answer is recorded with reasoning.

### Existing a11y attributes — must remain intact after listener fires
The cross-tab listener calls `setStats(loadStats())`, which triggers a React re-render of the same `<StatsFooter>` component with new props. The existing a11y surfaces in `StatsFooter.tsx` and surrounding components MUST survive that re-render unchanged:

- `<button type="button" name="Reset">` (StatsFooter Reset link) — `screen.getByRole('button', { name: 'Reset' })` must still match after the listener fires.
- `<div role="presentation">` on line 2 of the footer — unchanged.
- Footer visibility gating (`totalSessions > 0 && !inSessionView`) — unchanged. Listener-driven `totalSessions: 0` correctly re-hides the footer via existing gating.
- No new `aria-*` attributes are introduced. No existing `aria-*` attributes are removed or modified.

### Keyboard focus
**Decision:** A storage-event-driven re-render MUST NOT move keyboard focus. The footer re-renders the same DOM nodes via React reconciliation; React preserves focus on stable DOM nodes. The planner should verify that the `<button>` for Reset retains the same React key/identity across the re-render so focus is not lost if a screen-reader user has tabbed onto it when another tab writes. (Practically: `setStats` does not unmount the footer, so this is preserved by default — but the contract is locked here.)

### Mid-session listener firing
Per CONTEXT.md D-06a, the listener may fire while `inSessionView === true` (the footer is hidden via existing gating). In that case `setStats` updates React state but the user sees nothing because the footer is not rendered. Returning to the idle screen, the footer renders with the latest stats. No a11y impact — no announcement is missed because the user did not need one in the first place.

## Interaction Contract — STORAGE-03 Cross-Tab Stats Refresh

This is the **primary contract** Phase 8 adds. Locked verbatim from CONTEXT.md D-05/D-06/D-06a.

### Event source
- **Single trigger:** `window` `'storage'` event. No `focus`, no `visibilitychange`, no `BroadcastChannel`, no `setInterval` poll. (CONTEXT.md D-06.)

### Event filter
- Listener filters by `e.key === STATE_KEY` (the constant exported from `src/storage/storage.ts:35`, currently `'hrv:state:v1'`). Events for any other key are ignored. (CONTEXT.md D-06a.)
- No filtering by `e.newValue` content — even `e.newValue === null` (cleared localStorage) is allowed to fall through. (CONTEXT.md D-06.)

### Refresh scope
- **Stats only.** The listener calls `setStats(loadStats())` exactly. Settings and mute are NOT refreshed cross-tab — that drift is deferred to v1.1. (CONTEXT.md D-05.)
- `loadStats()` re-reads disk via `readEnvelope().stats → coerceStats(...)`. The cleared-storage edge case is handled naturally by `coerceStats(undefined) === ZERO_STATS`. (CONTEXT.md D-06.)

### Lifecycle
- Listener is attached **once at App mount** via `useEffect(() => { window.addEventListener('storage', handler); return () => window.removeEventListener('storage', handler); }, [])`. (CONTEXT.md D-06a.)
- Cleaned up on App unmount via the returned cleanup function.
- Empty dependency array (`[]`) — the handler closes over `setStats` (stable identity from `useState`) and the module-level `loadStats` import (stable). No `react-hooks/exhaustive-deps` violation against the Phase 7 strict-mode baseline.

### Mid-session tolerance
- The listener may fire at any time, including while `inSessionView === true` (lead-in or running). This is **explicitly allowed** because `setStats` is React-state-only — it triggers no domain side effects, no audio resets, no clock perturbation, no wake-lock churn. (CONTEXT.md D-06a.)
- The footer is hidden during `inSessionView` via existing D-10 gating, so the user sees no visual change mid-session. State update happens silently; footer reflects the refreshed value when the session view exits.

### Observable behavior in the running tab (the one that did NOT write)
When another tab writes the envelope and dispatches the `'storage'` event:
1. The listener fires synchronously on the next event-loop tick.
2. `loadStats()` reads the new envelope from disk (single `localStorage.getItem(STATE_KEY)` + JSON parse + `coerceStats`).
3. `setStats(newStats)` schedules a React re-render.
4. On the next React commit, the `<StatsFooter>` component receives the new `stats` prop.
5. The text nodes (`formatSessionCount`, `formatTotalMinutes`, `formatLastSession`) re-render with the new values.
6. **No flash. No transient state. No animation. No layout shift.** The numbers tick over to the new value on the next browser paint.
7. If the new envelope has `totalSessions === 0`, existing gating (`totalSessions > 0 && !inSessionView`) un-mounts the footer entirely — the user sees the footer disappear (e.g. after the other tab clicks Reset).

### Edge cases — locked behavior

| Edge case | Locked behavior | Rationale |
|-----------|------|-----------|
| `e.newValue === null` (other tab cleared localStorage) | `loadStats() → coerceStats(undefined) → ZERO_STATS` → footer hides via existing gating | CONTEXT.md D-06 — no explicit branch needed |
| `e.newValue` is malformed JSON | `readEnvelope` already swallows in `try/catch` and returns `EMPTY_ENVELOPE` → `coerceStats(undefined) → ZERO_STATS` → footer hides | Phase 4 D-17 silent-fallback posture (existing) |
| Same-tab write (no event fired for the writing tab per the StorageEvent spec) | Listener does not fire; existing in-tab `setStats` after `recordSession`/`resetStats` handles same-tab updates | Standards-defined behavior; no additional code needed |
| Event fires during App unmount | Cleanup function (`removeEventListener`) ran first; handler is not invoked | Standard `useEffect` cleanup pattern |
| Event fires after fast unmount/remount | New mount attaches a fresh listener with a fresh `setStats`; no stale closure | `useEffect([])` mount/unmount semantics |
| Listener fires while another `setStats` is in flight (e.g. mid-`confirmReset`) | React batches both updates; latest disk value wins on render commit | React 18 automatic batching — no special handling needed |

### Things this contract does NOT do (locked exclusions)
- Does NOT add a toast or banner saying "stats updated from another tab".
- Does NOT highlight or animate the changed numbers.
- Does NOT play an audio cue.
- Does NOT call any audio engine, wake lock, or session engine API.
- Does NOT refresh settings or mute (deferred to v1.1 per D-05).
- Does NOT fix the cross-tab `recordSession` increment race (deferred to v1.x debt per D-05 / CONTEXT.md / ROADMAP).
- Does NOT add a `console.warn` even in DEV when a refused-downgrade write occurs (D-03 silent-refusal posture).

## Strict-Mode Type Compliance (Phase 7 baseline)

Per CONTEXT.md "Phase 7 strict-mode baseline", every code example in this UI-SPEC must type-check cleanly under `strict: true` + `noUncheckedIndexedAccess: true` + `strictTypeChecked` ESLint. The interaction contract above implies the following type-level invariants for the executor:

- The `'storage'` event handler parameter is typed `(e: StorageEvent) => void` (DOM lib type, no widening).
- `e.key` is typed `string | null` (per DOM lib); the `e.key === STATE_KEY` comparison narrows it.
- `loadStats()` return type is `PersistedStats` (existing); `setStats` parameter type matches (existing).
- No `any`, no `as unknown as ...` casts in the listener body.
- The `useEffect` cleanup return is typed `() => void` (TypeScript infers it from `removeEventListener` return).
