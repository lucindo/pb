The cross-tab writer is technically untrusted (DevTools could be used to inject `StorageEvent` from the same origin), but the event is same-origin-bound and the listener only updates the existing stats footer text. |
| StorageEvent <-> handler payload | `e.key`, `e.newValue` are caller-controlled (the browser populates them from the other tab's setItem, but a same-origin script can dispatch synthetic events too). The handler reads `e.key` only to filter; `e.newValue` is NOT trusted — the handler re-reads disk via `loadStats()` rather than parsing the event payload. |
| inSessionView gating | Mid-session listener firings update React state but the footer is hidden via existing D-10 gating. The session engine, audio engine, and wake-lock subsystems are not invoked. |

| T-08-08 | Denial of Service | Cross-tab StorageEvent flood | mitigate | The handler does `setStats(loadStats())` — synchronous, O(1) localStorage read + JSON.parse + coerceStats. | T-08-09 | Tampering | StorageEvent.newValue ignored | mitigate | The handler reads `e.key` only. `e.newValue` is NOT parsed; instead `loadStats()` re-reads disk via `readEnvelope` which applies the same hardened parsing path as mount-time. A malicious dispatcher cannot inject crafted stats via the event payload — they would have to setItem first, and that path is protected by `readEnvelope`'s D-17 silent fallback (existing) plus `coerceStats` (existing). |
| T-08-10 | Tampering | StorageEvent.key spoof | accept | A same-origin script can dispatch `new StorageEvent('storage', { key: STATE_KEY, ... })` and trigger a `loadStats()` call. The result is a re-read from disk — no privilege escalation, no data corruption. Accepted: same-origin attacker can already setItem directly; the synthetic event adds no new capability. |

WCAG 2.2 SC 4.1.3 compliant. |
| T-08-12 | Denial of Service | In-progress session interruption | mitigate | `setStats` is React-state-only — calling it during a running session does NOT touch audio engine, wake lock, session clock, or boundary cue scheduling. | T-08-13 | Denial of Service | Listener fires after unmount | mitigate | The useEffect cleanup function (`removeEventListener`) runs at unmount before any subsequent dispatch can invoke the handler. Standard `useEffect` lifecycle — no stale-closure setStats call. RESEARCH §RQ-6 confirms via existing precedent in `useAudioCues.ts:176-179`. |

No `high` severity threats remain — all severe threats (T-08-09 untrusted payload parsing, T-08-12 session interruption, T-08-13 post-unmount fire) are mitigated by design. T-08-08 and T-08-10 are accepted as same-origin attacker model out-of-scope for ASVS L1 on a fix-only patch milestone.
