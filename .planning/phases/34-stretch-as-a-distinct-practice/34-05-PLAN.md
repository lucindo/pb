<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| localStorage → App state | `App.tsx` hydrates stretch settings/stats from the persisted envelope via `loadPractices()`; the storage-layer coercers (Plan 02) are the validation gate. |
| User input → SettingsForm → storage | Stretch ramp stepper values are bounded by the discrete option arrays (`STRETCH_INITIAL_BPM_OPTIONS`, etc.); only valid values can be selected. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-34-08 | Tampering | `App.tsx` hydrating `stretchSettings` from `initialPractices.stretch.settings` | mitigate | The value is already passed through `coerceStretchSettings` (Plan 02) at the storage boundary before reaching App state — App never reads the raw envelope directly. |
| T-34-09 | Denial of Service | stretch session engine path with a corrupt `StretchSettings` | mitigate | `buildStretchSegments`/`startStretchSession` operate on a coerced `StretchSettings`; per-field coercion guarantees every field is in-range, so the segment table is always finite — no infinite-loop or NaN-frame path. |
| T-34-10 | Tampering | `recordStretchSession` write at session end | mitigate | Slice-isolated write (Plan 02): spreads `...practices` and overrides only `stretch.stats` — a session-end write cannot corrupt resonant or naviKriya stats. |
</threat_model>
