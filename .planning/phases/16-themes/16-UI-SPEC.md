## Color

The codebase ships **one global accent token** (`--color-breathing-accent` = `#0f766e` teal-700 today) and a small set of `--color-*` tokens used by all components. Phase 16 introduces **5 visible palettes** by re-declaring those same token names inside `[data-theme='X']:root { ŌĆ” }` blocks. The 60/30/10 split is described **per palette** because the values rotate; the **token roles** stay fixed.

### Cross-Palette Token Role Map (the 60/30/10 contract ŌĆö applies to ALL 5 palettes)

| Role | % | Token(s) | Element |

### Accent reserved for (explicit list ŌĆö never "all interactive elements"):

The `--color-breathing-accent` token is reserved for:

1. **Selected-theme indicator** ŌĆö the option button currently matching `useThemeChoice().theme` paints its border with the accent ring (see "Selected indication" below).
2. **`focus-visible` ring** on every theme option button (`focus-visible:ring-2 focus-visible:ring-breathing-accent focus-visible:ring-offset-2` ŌĆö matches Phase 1 SettingsStepper pattern).
3. **Existing baseline reservations** (NOT introduced by Phase 16; inherited): `SettingsAnchor` gear glyph color when active, `LearnDialog` external-link text color, `StatsFooter` reset-action text color, primary CTA buttons in `SessionControls.tsx`. None of these are edited by Phase 16.

The accent token is NEVER applied to: non-selected option buttons, decorative borders, secondary text, the modal title, the modal Close button.

## Visual & Interaction Specifications

This section resolves the 9 discretion items from the orchestrator's `<discretion_areas_to_resolve>` block.

### 1. Picker UI shape (resolves discretion #1)

**Decision: Radio-like button group** with `role='radiogroup'` + `role='radio'` + `aria-checked` semantics. Six native `<button type="button">` elements inside a `<div role='radiogroup'>` container.

**Why this shape (over the 3 alternatives in RESEARCH.md ┬¦"Picker UI Shape Recommendation"):**

| Alternative | Why rejected |
|-------------|--------------|
| Native `<select>` | (a) Hides 5 of 6 options behind a click ŌĆö fails Phase 15 D-10 "all options visible at once" posture established by `SettingsStepper`. (b) Mobile OS chrome (Safari iOS, Chrome Android) renders cross-platform inconsistent dropdowns that clash with the modal's pastel aesthetic. (c) Doesn't match any other picker style in the dialog (Variant/Timbre/Language will be the same shape per the cross-picker consistency this UI-SPEC establishes). |
| Segmented control (6 buttons inline, `aria-pressed`) | (a) `aria-pressed` is a toggle semantic, not single-select; `role='radio'` is the correct ARIA mapping for "pick one of N". (b) 6 inline segments cram horizontally on mobile ŌĆö `max-w-md = 448px` leaves ~74px per segment after padding ŌĆö labels "System" and "Slate" abut their neighbors with no breathing room. Visual cohesion fails. |
| 6 distinct option cards (with miniature swatch previews) | (a) Adds ~150 LOC of swatch-rendering logic for one feature phase. (b) Forces the planner to lock the hex preview against the actual `--color-*` token at PLAN.md time, but the swatch-source preference (D-04) is "use CSS custom-property reference, not literal hex" ŌĆö implementable but invites visual flicker if the picker is rendered before the FOUC script lands. (c) Violates Phase 15 D-18 "minimal posture, no description copy" ŌĆö a visible color preview IS descriptive copy, even without words. |

**Chosen shape rationale:**
- 6 options visible at a glance ŌåÆ no surprise; matches `SettingsStepper`'s "options laid out, not hidden" posture.
- `role='radiogroup'` + `role='radio'` + `aria-checked` is the W3C-recommended ARIA pattern for single-select option groups [WAI-ARIA Authoring Practices ŌĆö Radio Group].
- Native `<button>` elements satisfy 44├Ś44 touch target floor (Phase 2 D-12) via `min-h-12` (48px) without any extra wrapper.
- Zero new dependencies (D-18) ŌĆö pure React + Tailwind.
- Visually coheres with the existing `SettingsDialog` Close button shape (`min-h-12 rounded-full border border-teal-200 bg-white px-5 py-2 text-base font-semibold text-teal-800`).

### 2. Option visual: text-only vs swatch vs mini-card (resolves discretion #2)

**Decision: Text label only.** No color chip, no mini-card preview.

**Rationale:**
- Phase 15 D-18 minimal-posture: "no description copy under options" ŌĆö and a visible color swatch IS visual description.
- Tracks the existing `SettingsStepper` value-output posture (`text-2xl font-semibold` numeric value only, no decorative chip).
- Avoids the swatch-source ambiguity: a `bg-[var(--color-breathing-bg-soft)]` chip would render the CURRENTLY-applied theme's hue inside every option button ŌĆö confusing (the "Dark" button's chip would show the user's currently-active palette, not dark's preview). Resolving this would require either (a) rendering a separate `<style>` per option with a scoped `[data-theme-preview='X']` attribute (~80 LOC), or (b) hardcoding hex literals into the picker (violates D-04 "hexes locked in PLAN.md, not in components").
- Reduces THEME-05 risk: no per-option mini-orb gradient that would also need contrast verification.
- The user sees the theme apply **instantly** on click (D-04 + S-01 ŌĆö the `data-theme` attribute swap is synchronous). Live preview > miniature preview.

**Future deferral:** A swatch variant can be added in v1.2+ as a polish phase; the planner can revisit if real-device UAT surfaces "I can't tell which one I want without trying it" friction.

### 3. Currently-selected indication (resolves discretion #3)

**Decision: `aria-checked='true'` + filled-background highlight + bordered ring.**

**Visual contract:**

| State | Border | Background | Text | ARIA |
|-------|--------|------------|------|------|
| Not selected, enabled | `border border-teal-200` | `bg-white` | `text-teal-800` | `aria-checked='false'` |
| Not selected, enabled, hover | `border border-teal-200` | `bg-teal-50` | `text-teal-800` | `aria-checked='false'` |
| Not selected, enabled, focus-visible | `border border-teal-200` + `ring-2 ring-breathing-accent ring-offset-2` | `bg-white` | `text-teal-800` | `aria-checked='false'` |
| **Selected, enabled** | `border-2 border-[var(--color-breathing-accent)]` | `bg-[var(--color-breathing-bg-soft)]` | `text-[var(--color-breathing-accent-strong)]` | `aria-checked='true'` |
| Disabled | (inherits non-selected styling) | (inherits) | (inherits) | `aria-checked` reflects current, `disabled` attr on `<button>`, `aria-disabled='true'` on the radiogroup |
| Selected + disabled | same as Selected, plus `opacity-45 cursor-not-allowed` | same | same | `aria-checked='true'` + `disabled` |

**Why filled background + ring (not just checkmark, not just outline):**
- A bare checkmark glyph fails the no-icon-library contract (Phase 15 D-15) and adds asymmetric layout to the option buttons (selected option becomes wider).
- A pure outline ring is too subtle on the pastel palette (the `--color-breathing-accent` ring on white is the SAME visual weight as the focus-visible ring ŌĆö users couldn't distinguish "selected" from "focused").
- Filled background (`--color-breathing-bg-soft`) + accent-colored border thickness step (1px ŌåÆ 2px) + accent-strong text gives a **3-channel signal** (background, border, text) that reads at any size and survives all 5 palettes (the token references re-tint per palette).
- Works for both keyboard nav (focus-visible adds the separate ring on top of the selected-state highlight) and pointer (hover state is only present for non-selected options; the selected option does not change on hover, signalling "you're already here").

**Reduced-motion contract:** the selected-state visual is INSTANT ŌĆö no fade-in, no scale-on-click animation. The CSS `transition` on these buttons is restricted to `transition` (background-color + border-color, 150ms) under normal motion, and `motion-reduce:transition-none` (matches existing `SettingsStepper.tsx:45` + `SettingsDialog.tsx:91` pattern) ŌĆö confirming discretion item #7 (transition behavior).

### 4. Disabled visual (resolves discretion #4)

**Decision: Mirror the `SettingsStepper.tsx:45` pattern verbatim.**

- `disabled` HTML attribute on each `<button>` (browser-native disable: ignores clicks, skips Tab focus).
- `aria-disabled='true'` on the `<div role='radiogroup'>` container (defense-in-depth ŌĆö Phase 6 LearnAnchor D-03 "disable not hide" carried).
- Visual: `disabled:cursor-not-allowed disabled:opacity-45` (Tailwind utility ŌĆö matches every other disabled button in the project: `SettingsStepper`, `MuteToggle`, `SettingsAnchor`, `LearnAnchor`).
- **The selected option keeps its highlight even while disabled** ŌĆö users see what's currently selected during a session, just can't change it.

This is fully consistent with RESEARCH.md ┬¦"Open Question 4 ŌĆö Visual styling of disabled state" recommendation.

### 5. Focus-visible / keyboard navigation (resolves discretion #5)

**Decision: Each option is its own tab stop (no arrow-key navigation).**

**Rationale (defers the deeper arrow-key polish):**
- The existing pickers (`SettingsAnchor`, `LearnAnchor`, all four Phase 15 stub pickers, `SettingsStepper`'s ŌłÆ / + buttons) all use plain Tab navigation. No component in the codebase implements `role='radiogroup'` arrow-key trapping.
- Implementing arrow-key navigation requires a `onKeyDown` handler at the radiogroup level intercepting `ArrowLeft`/`ArrowRight`/`ArrowUp`/`ArrowDown` and shifting focus via `ref.current?.focus()` ŌĆö ~25 LOC of new keyboard logic with edge cases (wrap-around at end-of-row vs end-of-group, RTL handling for Phase 19 PT-BR ŌĆö currently LTR-only but the I18N pipeline will need this re-verified).
- Native `<input type="radio">` would get arrow-key trapping for free, but using `<input>` with custom `<label>` styling fights Tailwind v4 + creates form-submission semantics that don't apply here.
- Consistency with the rest of the modal trumps the extra a11y polish for v1.1 ship.

**Keyboard contract for Phase 16 (locked):**
- `Tab` from the dialog title ŌåÆ enters the radiogroup ŌåÆ focuses the FIRST option (`Light`).
- `Tab` again ŌåÆ focuses the next option (`Dark`).
- ... through `Dusk`, then `Tab` leaves the radiogroup ŌåÆ focuses the next sibling picker (`VariantPicker`).
- `Shift+Tab` reverses.
- `Space` / `Enter` on a focused option ŌåÆ calls `setTheme(id)` (native `<button>` activation).
- `Escape` on any focused option ŌåÆ bubbles to the `<dialog>` cancel handler ŌåÆ closes the dialog (Phase 15 D-11 contract).
- `focus-visible` ring is `ring-2 ring-breathing-accent ring-offset-2` ŌĆö matches every other focusable element in the modal.

**Deferred polish (v1.2+):** If real-device UAT (informal ŌĆö Phase 16 has no formal UAT step per D-13) surfaces friction, add arrow-key navigation as a follow-up phase. Tracked in `Deferred Items` below.

### 7. Transition behavior (resolves discretion #7)

**Decision:**
- **Theme application (the `data-theme` attribute swap on `<html>`):** INSTANT. No CSS transition on any `--color-*` token. (Phase 16 deferred ideas: "Theme transition animation ŌĆö risks degrading THEME-05 perceptual contract during the in-flight tween. Locked out.")
- **Option button selected-state indicator:** 150ms `transition` on `background-color`, `border-color`, `color` under normal motion. Suppressed under `motion-reduce:transition-none` (matches `SettingsStepper.tsx:45` pattern).
- **Hover state on non-selected options:** 150ms `transition` to `bg-teal-50`. Same `motion-reduce:transition-none` gate.
- **Dialog backdrop fade (Phase 15 F3 ŌĆö locked, NOT edited):** 200ms opacity transition; suppressed under reduced motion.
- **No scale-on-hover, no translate-on-active.** Touch + click feedback is via the background-color shift only.

**Reduced-motion contract (preserves Phase 2 D-07 baseline):** All transitions on the theme picker are gated by `motion-reduce:transition-none`. Under `prefers-reduced-motion: reduce`, option selection produces an INSTANT visual swap with no fade. This matches the existing project posture; the orb crossfade contract (the only "essential" transition under reduced motion, per Phase 2 D-07) is unchanged by Phase 16.
