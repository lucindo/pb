# Phase 49.1: Code Review Report

**Reviewed:** 2026-05-27
**Depth:** standard
**Files Reviewed:** 29
**Status:** issues_found

## Summary

Phase 49.1 implements the Advanced page rename (formerly Appearance) and adds a
"Bypass silent mode" toggle as the fifth persisted feature flag. The
implementation follows the locked patterns from Phase 47/48 verbatim:
`BYPASS_SILENT_MODE_FLAG` mirrors `SWITCHER_ICON_FLAG`, `coerceBypassSilentMode`
mirrors `coerceSwitcherIcon`, `useBypassSilentModeChoice` mirrors
`useSwitcherIconChoice`, and the audio-engine gate at `audioEngine.ts:189`
correctly uses `!== false` so undefined coerces to "construct" (D-07 backward
compat). The rename is atomic — no leftover `'appearance'` literals or
`AppearancePage`/`onAppearanceOpen`/`returningFromAppearance` identifiers
remain in `src/`. PT-BR `// TODO: native-speaker review` markers were correctly
removed from flipped values (`title`, `rightChevronAriaOnSettings`, the
replaced `visual` → `behavior` key) and correctly omitted from net-new keys
(`bypassSilentMode.label`) per D-04.

Three quality concerns surfaced: (1) the public `start()` signature now widens
silently for callers that omit the 3rd arg, with no compile-time enforcement
that view-model callers actually thread the flag; (2) one test assertion is
structurally non-discriminating (`expect.objectContaining({ key: undefined })`
trivially passes when the key is absent); (3) an early-return path in
`useAudioCues.start` skips updating `bypassSilentModeRef`, which is benign
today but introduces an invariant gap if start-while-running semantics ever
change. No security, correctness, or data-loss issues identified.

## Warnings

### WR-01: `expect.objectContaining({ bypassSilentMode: undefined })` is non-discriminating

**File:** `src/hooks/useAudioCues.test.tsx:1287-1289`
**Issue:** The test asserts:

```ts
expect(createSpy).toHaveBeenCalledWith(
  expect.objectContaining({ bypassSilentMode: undefined }),
)
```

`expect.objectContaining` only checks the listed keys against the actual
object's enumerable properties, but matchers treat a missing key and an
explicit `undefined` value as equivalent at the matcher level. If a future
refactor removes the `bypassSilentMode` key from the object literal at
`useAudioCues.ts:254` (rather than passing it as `undefined`), this test
remains green even though the contract it claims to lock — "the hook must
forward undefined (not elide the key)" per the comment on line 1285-1286 —
is no longer enforced. The test does not discriminate the two cases.

**Fix:** Use `expect.objectContaining` to assert the key is present and
explicitly check the property's presence/value with a separate access on the
captured call, e.g.:

```ts
expect(createSpy).toHaveBeenCalledTimes(1)
const opts = createSpy.mock.calls[0]?.[0] as Record<string, unknown>
expect('bypassSilentMode' in opts).toBe(true)
expect(opts.bypassSilentMode).toBeUndefined()
```

This rejects both the "key absent" and the "key present with wrong value"
mutations — making the test discriminating in the sense the inline comment
already commits to.

### WR-02: `useAudioCues.start` early-return skips updating `bypassSilentModeRef`

**File:** `src/hooks/useAudioCues.ts:237-240`
**Issue:** When `start()` is called twice without an intervening `stop()`, the
defensive early-return at line 237-240 returns the cached `firstInCueTime`
without touching `timbreRef` or `bypassSilentModeRef`:

```ts
const existing = engineRef.current
if (existing !== null) {
  return firstInCueTimeRef.current
}
```

Today this is correct — the engine was constructed with the original session's
values and a mid-session toggle does NOT rebuild the engine (D-09 invariant) —
but the ref-update at line 253 (`bypassSilentModeRef.current = bypassSilentMode`)
is now *conditionally* gated on "engine doesn't exist yet". A future change
that decides to apply the new `bypassSilentMode` value on a re-`start()` (e.g.,
to retroactively patch the engine, or to support a `start()`-while-running
"refresh" semantic) would silently leave the ref stale, and the next
`reconstructEngine()` call would replay the *previous* session's value instead
of the most recent one.

**Fix:** Move the `timbreRef.current = timbre` and
`bypassSilentModeRef.current = bypassSilentMode` assignments to the top of
`start()`, BEFORE the `existing !== null` check. Today this is a no-op (the
values would just be overwritten with the same data), but it locks the
invariant "after `start()` returns, the refs always reflect the most recent
caller-provided values" — making future intent-changes discoverable through
the diff rather than through a silent stale-ref bug. Alternatively, document
the early-return as "engine is already running, the second call's args are
intentionally ignored" in the inline comment so the invariant is explicit.

### WR-03: `useBreathingSessionController.UseBreathingSessionControllerArgs.bypassSilentMode` is silently optional

**File:** `src/hooks/useBreathingSessionController.ts:58-63`
**Issue:** The new prop is declared as `bypassSilentMode?: boolean`. The
header comment justifies the optionality with "Until Plan 03 supplies the
value, undefined passes through to useAudioCues.start → audioEngine → D-07
undefined-coerces-to-true → Phase 49 v3 behavior preserved (acceptable
wave-1 → wave-2 gap state)". Plan 03 is in the same phase landing, so the
"gap state" rationale no longer applies at HEAD — there is exactly one
production caller (`useAppViewModel.ts:63`) and it always passes the flag.

Making the prop optional means:
1. A future caller (test, mock, integration harness) that forgets to thread
   the flag silently falls back to "construct silent loop" without any
   compile-time signal. The user-facing toggle has no effect for that caller.
2. The fallback is `undefined` → engine's `!== false` predicate → construct.
   The default-on posture is intentional for `useAppViewModel`'s call site
   (the flag default is `true`), but for a test or new caller this disguises
   the missing wiring.

Note: there is no test for `useBreathingSessionController` (verified via
grep — `useBreathingSessionController.test.{ts,tsx}` does not exist), so the
optional-prop slack has no current consumer. The risk is forward-only.

**Fix:** Make the prop required: `bypassSilentMode: boolean`. The single
production caller (`useAppViewModel.ts:63`) already passes the flag. The
"wave-1 → wave-2 gap state" header comment can be deleted at the same time
since wave 2 is now landed. Required-prop forces any new caller (test or
production) to make an explicit choice rather than silently inheriting the
construct-loop default. (Alternatively: keep the prop optional but document
the contract change in the header comment and add a runtime assertion
mid-render that flags accidental omission — heavier touch.)

### IN-05: Test file uses raw `'hrv:state:v1'` storage-key literal

**File:** `src/hooks/useAudioCues.test.tsx:1220-1222`
**Issue:** The reconstruction-timbre test seeds localStorage with the
literal key string `'hrv:state:v1'` instead of importing `STATE_KEY` from
`../storage`:

```ts
window.localStorage.setItem(
  'hrv:state:v1',
  JSON.stringify({ version: 1, prefs: { theme: 'system', timbre: 'flute', locale: 'en' } }),
)
```

The `storage.ts` header comment (lines 31-32) explicitly says "Tests should
NOT depend on the literal STATE_KEY string; assert through the public
load*/save* API where possible. Tests that DO use STATE_KEY directly
[should import it]". The seeded prefs envelope is also incomplete (missing
`cue`, `breathingShape`, `ringCue`, `orbIdle`, `switcherIcon`, and the new
`bypassSilentMode`) — `coercePrefs` will fill in defaults per field, so the
test still proves what it claims (timbreRef stays `'bell'` across reconstruct),
but it is a brittle adversarial input and inconsistent with the convention
used elsewhere in the test file (the bypass tests at lines 1294-1321 use the
public `start()` API and never touch localStorage).

This test pre-existed Phase 49.1 (it is the Phase 18 D-11 timbre-invariant
guard); this phase did not introduce the smell, but the new tests in the
same file inherit the convention divergence. Worth flagging during this
phase's review so the next maintenance pass can converge.

**Fix:** Import `STATE_KEY` from `../storage` at the top of the file and use
it in the literal. Extend the seeded prefs to a full 9-field UserPrefs (or
spread `DEFAULT_PREFS`) to make the test's adversarial-input contract
explicit.
