# Phase 49.1 ŌĆö Advanced Settings page + "Bypass silent mode" toggle

## Why this phase exists

Phase 49 shipped the silent-loop `<audio>` element always-on (D-02 lock ŌĆö
"the silent loop is always on; UA-gating only if regression appears").
Device validation showed it works (iOS speaker routing closed), and code
review caught one real leak (CR-01, fixed). But the operator reframed the
feature scope after validation: **bypassing the physical silent switch is a
user-facing behavior choice**, not a transparent platform workaround. Some
users may legitimately prefer the silent switch to silence the app (the iOS
"ambient" default); others want the bypass.

The Phase 48 infrastructure (persisted feature flags + query-string parity +
SettingsToggleRow + 2-arg `readFeatureFlags(search, persisted)` resolver)
already exists. Phase 49.1 piggybacks on it to surface the silent-loop as a
user toggle, default-on (preserve Phase 49 shipped behavior for users who
don't touch the setting). The existing Appearance page (Phase 48) is the
natural home, but its name is too narrow once it carries a non-visual
behavior toggle ŌĆö operator decision is to rename it to **Advanced**.

- **D-03:** New toggle is added **below** the existing two Behavior-section

- **D-04 (i18n):** All EN + PT-BR strings introduced by this phase
  (`advanced.title` / `Avan├¦ado`, `advanced.sections.behavior` /
  `Comportamento`, `advanced.bypassSilentMode.label` / `Ignorar modo
  silencioso`) are **operator-confirmed native-validated**. **No** `// TODO:

### Behavior + defaults

- **D-05:** New persisted feature flag: **`bypassSilentMode: boolean`**,
  **default `true`** (preserves Phase 49 shipped behavior for existing users
  who never touch the toggle). Stored in the same `featureFlags` blob as
  the Phase 48 flags (`breathingShape` / `ringCue` / `orbIdle` /
  `switcherIcon`). Same per-field corrupt-field fallback pattern (Phase 8
  D-01) ŌĆö **no `STATE_VERSION` bump**.
- **D-06 (query-string parity):** `?bypassSilentMode=false` works exactly
  like the existing four flags via the Phase 48 2-arg `readFeatureFlags(search,
  persisted)` resolver. Precedence chain (query > persisted > default)
  unchanged. A new `BYPASS_SILENT_MODE_FLAG.parse` non-throwing coercer is
  added to match the existing flag-coercer pattern.
- **D-07 (audio gating):** `createAudioEngine(opts)` accepts a new
  `opts.bypassSilentMode?: boolean` field.
  - When **`opts.bypassSilentMode === false`**: skip silent-loop element
    construction entirely. No `new Audio(...)`, no `.play()` call, no close()
    teardown, no CR-01 resume-reject teardown (the existing
    `if (silentLoopElement !== null)` guards in close() and the
    resume-reject catch already handle the null case ŌĆö no new branches
    needed).
  - When **`opts.bypassSilentMode === true` OR `undefined`**: identical to
    Phase 49 shipped behavior (silent loop active). Undefined ŌåÆ true default
    keeps any call site that hasn't been updated working with the shipped
    Phase 49 posture.
- **D-08 (hook plumbing):** `useAudioCues.ts` reads `bypassSilentMode` from
  the same flag resolver the rest of the app uses, passes it through to
  `createAudioEngine({ ..., bypassSilentMode })` and to
  `reconstructEngine()`. **Phase 49's D-04 NO-CHANGE lock was scoped to
  Phase 49 only**; Phase 49.1 explicitly modifies `useAudioCues.ts` and
  `useAudioCues.test.tsx` to thread the flag. This deviation is documented
  here as the phase's new contract.
- **D-09 (live toggle behavior):** Changing the toggle mid-session does NOT
  rebuild the engine. The flag is read at engine construction time (Start
  click / `reconstructEngine()`); a toggle change applies on the **next**
  engine construction. UX rationale: rebuilding the audio engine on a
  preference toggle would interrupt the current session, which is worse
  than waiting for the next session start. Documented inline near the
  toggle.

### Internal naming

- **D-10 (route + enum rename):** `AppScreen` enum value `'appearance'` is
  renamed to `'advanced'` everywhere it appears (`useAppNavigation.ts`,
  `ScreenRouter.tsx`, `appControllerAdapters.test.ts`, etc.). The file
  `src/app/pages/AppearancePage.tsx` is renamed to `AdvancedPage.tsx`,
  along with its test file and the exported component name. The
  `useUiStrings().appearance` tree is renamed to `useUiStrings().advanced`
  end-to-end. **There is no backward-compatibility shim** ŌĆö this is an
  internal-API-only rename. The persisted flag keys (`breathingShape` /
  `ringCue` / `orbIdle` / `switcherIcon` / new `bypassSilentMode`) keep
  their existing names ŌĆö only the strings-tree + page + route enum change.
- **D-11 (atomic rename commit):** The AppearanceŌåÆAdvanced rename lands as
  a single commit (or a single executor task) so the per-commit green-gate
  (`tsc && lint && build && test`) holds at every commit. Splitting the
  rename across commits would temporarily break either the type imports or
  the strings lookup.

## Success criteria (falsifiable)

1. **ADV-01:** App Settings ŌåÆ trailing chevron lands on a page whose title
   reads "Advanced" (EN) / "Avan├¦ado" (PT-BR). The old string "Appearance" /
   "Apar├¬ncia" is **not present** anywhere in `src/content/strings.ts`.
2. **ADV-02:** The Behavior section on the Advanced page has the header
   "Behavior" (EN) / "Comportamento" (PT-BR). The old key
   `advanced.sections.visual` is **not present**; the new key
   `advanced.sections.behavior` is present.
3. **ADV-03:** A "Bypass silent mode" / "Ignorar modo silencioso" toggle is
   visible on the Advanced page, below the two existing Behavior toggles.
   It is checked by default for a fresh storage state. Toggling it persists
   across reloads.
4. **ADV-04:** When the toggle is OFF and the user starts a new session,
   `new Audio(SILENT_WAV_DATA_URL)` is **not constructed** (verified by
   stubbing global `Audio` and asserting the constructor is never called).
   When the toggle is ON (or unset), the silent-loop element is constructed
   exactly as it is in Phase 49 v3.
5. **ADV-05:** `?bypassSilentMode=false` query string overrides persisted
   storage for the current page-load lifecycle, identical to the precedence
   of the other Phase 48 flags. A user with persisted `true` who loads the
   page with `?bypassSilentMode=false` gets the bypass disabled for that
   session.
