<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → App.tsx | `beforeinstallprompt` event (captured by `useBeforeInstallPrompt`) and `isIOS`/`isStandalone` (from `useIsStandaloneOrPhone`) are browser-controlled inputs, surfaced as plain booleans. |
| App.tsx → SettingsDialog | `isIOS`, `isStandalone`, `installable`, `onInstall` cross as React props — internal, trusted, no user-text payload. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29-03 | Spoofing / Tampering | `installable` derived boolean | accept | `installable = isIOS || deferredPrompt !== null` is a pure derived boolean computed in App.tsx from browser-owned events. No external mutation surface — identical posture to the Phase 28 `showBanner` gate (T-28-08). The worst case of a wrong value is a visible-but-inert install row, which `onInstall`/iOS-steps handle gracefully (D-07: never a dead/broken button). |
| T-29-04 | Elevation of privilege | `onInstall` (triggerInstall) prop | accept | `triggerInstall` replays the browser-captured install prompt; the browser, not the app, gates the actual install. No app-side privilege boundary. The button is `disabled` during `inSessionView` (defence-in-depth alongside WR-09 auto-close). |
| T-29-05 | Information disclosure | PT-BR install copy in strings.ts | accept | Static localized UI copy; no PII, no secrets. Rendered as React text content (auto-escaped); no `dangerouslySetInnerHTML`. No new `localStorage` keys introduced by this plan. |
| T-29-06 | Denial of service | iOS steps inline-expand state | accept | `iosExpanded` is component-local boolean state toggled only by the user's own click. No unbounded growth, no external trigger. |

No high-severity threats. UI-only composition phase: no server, no auth, no user-text input, no new storage — same threat posture as Phase 28.
</threat_model>
