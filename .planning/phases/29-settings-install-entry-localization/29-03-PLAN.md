<objective>
Close GAP-1 from 29-VERIFICATION.md: the iOS install-step instructions rendered by the shared `IosInstallSteps` component are unreadable on both dark themes (`dark`, `dusk`).

Root cause (confirmed by reading `src/components/IosInstallSteps.tsx`): the first `<li>` (step 1) carries `className="text-[var(--color-breathing-accent-strong)]"`, but the second and third `<li>` elements (`{strings.iosStep2}` and `{strings.iosStep3}`) have NO text-color class at all. They inherit the page default text color, which is near-black — invisible against the dark-theme backgrounds (`dark` bg `#2e3440`, `dusk` bg `#191724`). The code review's "inconsistent color class on only the first `<li>`" finding is the same defect: step 1 is the only item with an explicit token.

The intended design is already documented in `29-UI-SPEC.md` (§Color, lines 83-85 and 96-98): step 1 uses `--color-breathing-accent-strong` (first-step highlight); steps 2 and 3 use `--color-breathing-muted` (iOS step 2 & 3 text). The `--color-breathing-muted` token is theme-aware and has verified WCAG AA contrast on every theme (`theme.css` records dark muted-vs-bg = 5.36, clearing the 4.5 body floor; light/moss/slate clear comfortably). Applying it to steps 2 and 3 fixes the contrast on all 5 themes in one change.

Because `IosInstallSteps` is a single shared component (D-06), this one fix corrects readability on BOTH the `InstallBanner` surface and the `SettingsDialog` install row simultaneously.

Purpose: restore INSTALL-06 acceptance — the iOS install path must be readable on every theme the operator can select.
Output: theme-aware text color on all three iOS step `<li>` elements, plus a regression test that pins the token classes.
</objective>
