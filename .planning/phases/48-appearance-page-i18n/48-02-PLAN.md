<objective>
Extend the in-memory router (`useAppNavigation.ts`) with a fourth `AppScreen` value (`'appearance'`), two new callbacks (`onAppearanceOpen`, `onBackToAppSettings`), and a `returningFromAppearance` sentinel for focus restoration on return to App Settings. Propagate the three new fields through the view-model surface (`AppDialogsViewModel` in `appViewModel.ts` + `createAppDialogsViewModel` in `appControllerAdapters.ts`). Extend `useAppNavigation.test.tsx` with the four transition assertions specified in D-16.

Purpose: This is the routing-state foundation Plan 04 binds to when it adds the Appearance page surface, the App Settings trailing chevron, and the conditional mount-focus on App Settings. The sentinel contract (set on back, cleared on every other transition) is what makes `[[feedback_design_logic_separation]]` honourable — the focus restoration touches state-machine wiring only, not engine/audio/persistence.

Output: Extended union + two callbacks + one sentinel field on the navigation hook; three new fields on `AppDialogsViewModel`; matching propagation in the adapter; four new transition tests.
</objective>

Sentinel-clearing rule (RESEARCH §Pitfall 4): every transition OTHER THAN `onBackToAppSettings` must `setReturningFromAppearance(false)`. The `closeOnSessionView` effect that already resets to `'practice'` must ALSO clear the sentinel. If only the `useEffect` consumer clears it, the user can re-enter Settings via a non-back path and still get chevron focus — a real UX bug.

Sentinel storage choice (D-13 + RESEARCH OQ#2): use `useState<boolean>` (simpler than `useRef`; one extra render is acceptable). Open question OQ#2 — `useRef` is an allowed refinement if the verifier flags the extra render; ship `useState` first.
