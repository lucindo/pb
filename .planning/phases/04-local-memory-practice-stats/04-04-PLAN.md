  <name>Task 1: Manual UAT — LOCL-01 cross-reload restoration (settings + mute)</name>
  <files>(none — manual verification only; produces a SUMMARY entry)</files>
  <read_first>
    - .planning/phases/04-local-memory-practice-stats/04-VALIDATION.md (Manual-Only Verifications table — first row)

  </read_first>
  <what-built>
    Plans 01-03 wired Phase 4 persistence: a versioned localStorage envelope (`hrv:state:v1`), per-field validate-and-fallback restore on mount, and persisted setters that write on every BPM/ratio/duration/mute change. The integration suite proves this works under jsdom; this checkpoint verifies the same in a real browser across an actual page reload.
  </what-built>
  <how-to-verify>
    1. Start the dev server: `npm run dev` (Vite serves at http://localhost:5173 by default).
    2. Open the app in any modern browser (Chrome, Firefox, or Safari — NOT Private mode).
    3. From the default state, change ALL four persisted values:
       - BPM: change from `5.5` to a different value (e.g. `4` via the BPM stepper).
       - Ratio: change from `40:60` to a different value (e.g. `50:50`).
       - Duration: change from `10 min` to a different value (e.g. `20 min` or `open-ended`).
       - Mute: click the speaker icon to toggle mute ON (icon switches to crossed-out state, aria-label becomes "Unmute audio cues").
    4. Open browser DevTools -> Application/Storage -> Local Storage -> http://localhost:5173 and confirm there is a key `hrv:state:v1` whose JSON value contains all four updated fields.
    5. Close the tab entirely (Cmd-W / Ctrl-W).
    6. Reopen the app at http://localhost:5173 in a new tab.
    7. Verify ALL FOUR values are restored:
       - BPM stepper shows the value you set (e.g. `4`).
       - Ratio stepper shows the value you set (e.g. `50:50`).
       - Duration stepper shows the value you set (e.g. `20 min`).
       - Mute toggle is in the muted state you left it (aria-label "Unmute audio cues").
    8. Open DevTools -> Application -> Local Storage and confirm the envelope still contains your values.

    Expected outcome: all four values restored exactly. No banner, no toast, no error in the console other than possibly a benign React DevTools notice.

    If any value did NOT restore: the silent-fallback path may have triggered (corrupt JSON, quota error). Open DevTools console and report any thrown errors. This is a FAIL.
  </how-to-verify>
  <resume-signal>Type "approved" if all four values restored; describe any issue (which value drifted to default, console errors, etc.) if not.</resume-signal>
</task>

<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 4: Manual UAT — D-16/D-17 silent-fallback under Safari Private Browsing</name>
  <files>(none — manual verification; produces a SUMMARY entry)</files>
  <read_first>
    - .planning/phases/04-local-memory-practice-stats/04-VALIDATION.md (Manual-Only Verifications fourth row)
    - .planning/phases/04-local-memory-practice-stats/04-CONTEXT.md (D-16 silent absorb on write failure; D-17 silent absorb on read failure)
    - .planning/phases/04-local-memory-practice-stats/04-RESEARCH.md (Standard Stack table — Safari ITP / Private Browsing throw behavior)
  </read_first>
  <what-built>
    The storage adapter in `src/storage/storage.ts` wraps every `getItem` / `setItem` in `try { } catch` and silently absorbs ALL throws (D-16 / D-17). The app continues with in-memory defaults, with NO banner, NO toast, NO inline notice (matches Phase 3 D-10's silent-fallback posture).
  </what-built>
  <how-to-verify>
    Two checks: read-failure silent fallback AND write-failure silent fallback.

    **Check 1 — Read failure / first visit in Private Browsing:**
    1. If Safari is available: open Safari -> File -> New Private Window. Navigate to http://localhost:5173.
       - If Safari is not available, simulate via Chrome DevTools: open Chrome DevTools -> Application -> Storage -> check "Disable storage" or use an Incognito window with a manually broken localStorage (next paragraph).
    2. Verify the app loads cleanly with defaults: 5.5 BPM, 40:60, 10 min, mute off. NO banner. NO toast. NO inline error.
    3. Open the DevTools console. Verify there is no user-facing error message rendered (a `console.warn` from the dev gate is acceptable per D-17 but must not affect UI).

    **Check 1 (alternative) — Corrupt JSON injection:**
    1. In a normal (non-private) browser tab on http://localhost:5173, open DevTools console.
    2. Run: `localStorage.setItem('hrv:state:v1', '{not-json')` to inject a corrupt envelope.
    3. Reload the page.
    4. Verify the app still loads with defaults. NO banner. NO toast. NO error overlay. The corrupt envelope is silently swallowed (D-17).

    **Check 2 — Write failure / quota exceeded:**
    Quota exceeded is hard to reproduce in a browser without a separate test harness. The unit-test suite (`src/storage/storage.test.ts`) covers this via `vi.spyOn(Storage.prototype, 'setItem')`. For a manual smoke check, the read-side covered above is sufficient evidence that the silent-fallback wrapper is wired through. Optional deeper check:

    1. Open DevTools console.
    2. Run: `Object.defineProperty(Storage.prototype, 'setItem', { value: () => { throw new Error('quota') } })` to monkey-patch setItem to throw.
    3. Interact with the app: change BPM, toggle mute, complete a session.
    4. Verify NO banner, NO toast, NO error overlay. The app continues to function in-memory; persistence simply doesn't happen.
    5. Reload the page. Settings + stats revert to whatever was in the envelope before the patch (or defaults if nothing was there). This is the expected D-16 silent-fallback outcome.
    6. Refresh again to clear the monkey-patch (it lives only on the page).

    Expected outcome: under both Private Browsing AND injected-corrupt-JSON AND injected-quota-throw, the app shows defaults, never throws to the user, never displays a banner or toast.
  </how-to-verify>
  <resume-signal>Type "approved" if both read-failure and (optionally) write-failure paths are silent. Describe any issue (e.g. "uncaught exception in console blocked the page", "a red banner appeared saying 'Storage unavailable'", "the app crashed with a white screen").</resume-signal>
